# 卡片刷流状态门控设计

## 背景

当前工作区卡片在打开对话面板后，会持续接收 `session_messages_appended` websocket 增量消息。消息进入后会更新消息列表，`workspace-conversation-pane` 在 `messages` 变化时会自动滚动到底部。

这在 `running` 工作区里是合理的，但对已进入 `idle` 或 `attention` 的卡片会造成两个问题：

1. 小卡片或预览卡片会发生无意义的滚动和刷新。
2. 当最后一条消息已经结束后，仍继续接收后续 websocket 增量，前端缺少“停止刷流”的判定。

## 目标

增加前端卡片级实时消息接收策略：

- 当前工作区状态为 `running` 时，继续接收 websocket 增量消息。
- 工作区进入 `idle` 或 `attention` 后，如果当前最后一条消息已经处于终态，则不再接收新的 websocket 增量刷新。
- 工作区后续再次回到 `running` 时，自动恢复消息接收。

## 非目标

- 不修改后端 websocket 协议。
- 不重构现有 websocket 连接模型。
- 不单独调整滚动行为；以“阻断无效消息接收”作为主修复手段。

## 方案

### 1. 在前端实时层新增门控判断

在 `src/web/workspace-home.ts` 中增加卡片级判定函数，用于回答“当前工作区是否还应继续接收实时增量消息”。

建议规则：

- `workspace.status === "running"`：允许接收
- 否则读取当前卡片最后一条对话消息
- 若最后一条消息为终态，则拒绝接收新的增量消息
- 若最后一条消息仍处于流式过程，则继续接收，直到进入终态

### 2. 终态判定基于现有前端消息模型

不新增后端字段，直接复用前端消息归一化结果：

- `tool` / `tool-group` 的 `status` 为 `success`、`error`、`denied`、`idle` 时视为终态
- `running`、`pending` 视为非终态
- 普通文本消息不单独作为“结束标记”，避免误伤尚未完全收束的会话

这样可以兼容用户描述里的 `finished/completed` 语义，并映射到前端已有的工具状态汇总。

### 3. 门控接入 appendRealtimeMessages

在 `appendRealtimeMessages(sessionId, messages)` 中：

1. 先解析 `sessionId` 对应的 workspace
2. 调用门控判断
3. 若不允许接收，则直接返回
4. 若允许接收，再执行现有 merge / sort / group 流程

这样能把无效刷新挡在状态层，不需要改 `workspace-conversation-pane` 的滚动实现。

### 4. 重新开启接收依赖现有 workspace_snapshot

不单独维护“关闭/开启订阅”标记：

- `workspace_snapshot` 已经会同步最新工作区状态
- 当某卡片从 `idle/attention` 回到 `running` 时，门控函数会自动重新放行
- 这样可以避免额外状态同步成本

## 影响范围

- `src/web/workspace-home.ts`
- `tests/workspace-home.test.ts`

## 验证重点

1. `running` 工作区持续接收 websocket 增量消息
2. `idle/attention` 工作区且最后一条消息为终态时，忽略后续 append
3. `idle/attention` 工作区但最后一条消息未终态时，仍继续接收
4. 工作区状态重新变为 `running` 后，append 再次生效
