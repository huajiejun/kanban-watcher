# 2026-03-26-工作区前端端口分配设计

## 背景

当前 `scripts/start-dev.sh` 通过 `worktree_id` 哈希推导前后端端口。这个规则不再满足当前需求：

1. 前端端口必须限制在 `6020-6030`。
2. 前端启动时应通过接口申请可用端口，而不是脚本本地计算。
3. 端口状态要与工作区生命周期关联，工作区归档后可释放端口。
4. 后端端口与前端端口保持固定映射关系，避免额外维护成本。

## 目标

1. 前端端口只从 `6020-6030` 范围内分配。
2. 启动脚本通过后端接口获取端口并启动服务。
3. 同一未归档工作区重复启动时，优先复用已记录端口。
4. 后端端口通过 `frontend_port + 10000` 推导。
5. 端口信息落在当前项目数据库中，不新增独立映射表。

## 非目标

1. 不实现跨项目共享的通用端口服务。
2. 不支持单个工作区同时运行多个前端实例。
3. 不在本次重构现有后端启动流程，只让后端端口从前端端口推导。

## 方案结论

采用以下方案：

1. 在 `kw_workspaces` 上新增 `frontend_port INT NULL` 字段。
2. 启动脚本调用 `POST /api/workspace/{workspace_id}/frontend-port` 获取前端端口。
3. 接口负责分配、复用、失效检测与回写 `frontend_port`。
4. 后端端口不落库，统一按 `frontend_port + 10000` 推导。
5. 端口映射默认在工作区未归档期间保留；当工作区归档时清空字段释放端口。

## 数据模型

### `kw_workspaces`

新增字段：

- `frontend_port INT NULL`

语义：

1. `NULL` 表示当前未声明前端端口。
2. 非空表示当前工作区的稳定前端端口映射。
3. 字段只保存当前状态，不保存历史记录。

对应 `Workspace` 模型增加：

- `FrontendPort *int`

## 端口规则

### 前端端口

- 分配池固定为 `6020-6030`
- 首次启动时从池中挑选一个可用端口
- 同一工作区再次启动时优先复用原端口

### 后端端口

- 规则固定为 `frontend_port + 10000`
- 例如：
  - 前端 `6020` -> 后端 `16020`
  - 前端 `6027` -> 后端 `16027`

后端端口不单独存储，避免重复状态。

## API 设计

### `POST /api/workspace/{workspace_id}/frontend-port`

用途：为工作区获取可用前端端口，并返回对应的后端端口。

返回示例：

```json
{
  "success": true,
  "data": {
    "workspace_id": "ws-1",
    "frontend_port": 6023,
    "backend_port": 16023
  }
}
```

### 接口行为

1. 校验工作区存在且未归档。
2. 若 `frontend_port` 有值：
   - 若端口当前可用，则直接复用。
   - 若端口失效或已被其他进程占用，则重新分配。
3. 若 `frontend_port` 为空：
   - 在 `6020-6030` 中选择可用端口。
   - 将该端口写回 `kw_workspaces.frontend_port`。
4. 返回 `frontend_port` 及推导出的 `backend_port`。

## 启动脚本职责

`scripts/start-dev.sh` 后续改为：

1. 解析当前工作区 ID。
2. 调用接口获取前端端口。
3. 根据 `frontend_port + 10000` 计算后端端口。
4. 用这两个端口启动前后端。

脚本不再负责：

1. 计算前端端口
2. 扫描端口池
3. 维护端口映射状态

## 释放策略

### 默认策略

停止前端时，不立即清空 `frontend_port`。这样可以保持同一工作区的访问地址稳定。

### 释放时机

1. 工作区进入 `archived=true`
2. 现有端口被判定失效并被重新分配

这样可以兼顾稳定入口与端口回收。

## 并发控制

端口池只有 11 个端口，必须避免并发冲突。

建议在事务内完成：

1. 锁定目标工作区记录。
2. 读取所有未归档工作区已占用的 `frontend_port`。
3. 检查系统实际端口占用。
4. 选择空闲端口并写回数据库。

若池耗尽，接口返回明确业务错误。

## 失效判定

以下情况视为端口失效：

1. `kw_workspaces.frontend_port` 有值，但该端口当前已被其他无关进程占用。
2. 数据库端口值超出 `6020-6030` 合法范围。

失效时不直接失败，而是尝试重新分配。

## 错误处理

需要覆盖：

1. 工作区不存在
2. 工作区已归档
3. 前端端口池耗尽
4. 数据库记录端口失效且没有替代端口

接口需返回可读错误信息，方便脚本直接输出。

## 测试策略

### Store / Service

1. 首次申请端口成功
2. 同一工作区重复申请复用同一端口
3. 已记录端口失效后重新分配
4. 端口池耗尽时返回错误
5. 归档工作区清空端口

### API

1. `POST /frontend-port` 返回端口成功
2. 已归档工作区申请端口被拒绝
3. 返回值中 `backend_port` 等于 `frontend_port + 10000`

### 脚本

1. 启动脚本通过接口获取前端端口
2. 启动脚本按规则推导后端端口
3. 启动脚本不再本地计算前端端口池

4. 修改 `scripts/start-dev.sh` 调用接口并使用返回端口
5. 在归档流程中清空 `frontend_port`
