# 工作区远端布局同步设计

日期：2026-03-25

## 目标

为网页版工作区增加“本地与远端同步”的共享布局能力，使同一服务在不同端口打开时共享同一份布局状态，并通过 websocket 实时同步到所有在线端；同时保留“需要注意自动弹出”和“用户手动移走后当前轮次不再自动弹出”的交互语义。

## 已确认需求

- 工作区布局需要在本地和远端保持同步
- 远端增加，本地也增加；本地减少，远端也减少
- 前后端交互希望是双向的，服务端一端变化，多端同时同步
- 前端允许继续保留本地缓存，但远端需要成为跨端共享来源
- 需要注意任务在第一次进入 attention 时自动弹出
- 用户手动移走后，同一轮 attention 不再自动弹出
- 若任务离开 attention 后再次进入 attention，应视为新一轮 attention，可以重新自动弹出

## 现状

- 前端仅在 `localStorage` 保存 `openWorkspaceIds` 与 `activeWorkspaceId`
- attention 自动弹出的抑制状态仅存在当前浏览器内存中
- 后端尚无工作区布局读写接口
- 后端已存在 MySQL 存储层和 `/api/workspaces/active` 等工作区接口，可在同一 API 体系内扩展

## 方案对比

### 方案一：REST 持久化 + WebSocket 广播

前端启动先读本地做快速恢复，再通过 HTTP 拉取远端布局并覆盖本地；任何布局变更都先更新当前界面，再通过 HTTP 写远端，服务端写库成功后通过现有 websocket 广播最新共享布局，所有在线端同时收敛。

优点：

- 跨端一致性最好
- 兼顾秒开和远端共享
- 现有本地恢复链路可以继续复用
- 现有 realtime hub 可直接扩展，无需再造第二套长连接协议

缺点：

- 需要新增后端表和接口
- 需要处理首次远端覆盖本地的时序
- 需要补充广播事件与最终一致性规则

### 方案二：纯 WebSocket 双向协议

优点：

- 所有状态都走单一连接，实时感最强

缺点：

- 需要新增客户端发送指令、服务端 ack、重连补快照、顺序控制
- 明显超出当前代码已有能力，改动面更大

### 方案三：轮询远端布局 + 现有 websocket 保持不变

优点：

- 改动最少

缺点：

- 多端同步存在明显延迟
- 不符合“保持最终一致性，一端改变，多端同时同步”的目标

## 选型

采用方案一：REST 持久化 + WebSocket 广播。

## 数据模型

后端新增一张布局表，例如 `kw_workspace_views`，保存整服务共享的单例布局记录：

- `scope_key`
- `open_workspace_ids_json`
- `active_workspace_id`
- `dismissed_attention_ids_json`
- `version`
- `updated_at`
- `created_at`

其中：

- `open_workspace_ids_json` 保存当前右侧 pane 顺序
- `dismissed_attention_ids_json` 保存当前 attention 轮次内被用户手动移走、不再自动弹出的工作区 ID 集合
- `scope_key` 固定为 `global`
- `version` 为单调递增版本号，用于前端调试和最终收敛观察；冲突策略仍采用服务端最后一次成功写入为准

前端页面状态扩展为：

- `openWorkspaceIds`
- `activeWorkspaceId`
- `dismissedAttentionIds`
- `previousAttentionMap`
- `hasHydratedAttentionSnapshot`

其中：

- `dismissedAttentionIds` 持久化到本地和远端
- `previousAttentionMap` 和 `hasHydratedAttentionSnapshot` 仍仅存在于当前页面内存，用来判定“新进入 attention”的跃迁
- “当前轮次”的定义继续由前端基于 `previousAttentionMap` 和最新工作区快照判断；任务离开 attention 后，从 `dismissedAttentionIds` 中清除，下一次重新进入 attention 时允许再次自动弹出

## 接口设计

新增两个 HTTP API：

### `GET /api/workspace-view`

返回：

- `open_workspace_ids`
- `active_workspace_id`
- `dismissed_attention_ids`
- `version`
- `updated_at`

无记录时返回空布局对象，不报错。

### `PUT /api/workspace-view`

请求体：

- `open_workspace_ids`
- `active_workspace_id`
- `dismissed_attention_ids`

语义：

- 以整份布局覆盖远端当前记录
- 服务端负责做基础字段校验和 JSON 持久化
- 写入成功后返回规范化后的完整布局对象

## 实时事件设计

复用现有 `/api/realtime/ws`，新增事件类型：

### `workspace_view_updated`

负载：

- `open_workspace_ids`
- `active_workspace_id`
- `dismissed_attention_ids`
- `version`
- `updated_at`

服务端在 `PUT /api/workspace-view` 成功写库后立即广播该事件。

广播规则：

- 任一端提交布局成功后，所有在线端都收到同一份最新布局
- 当前发起写入的端也会收到广播，并以服务端返回内容为准重新收敛
- websocket 断线期间错过的事件，通过重新执行 `GET /api/workspace-view` 补快照

## 前端同步规则

### 启动恢复

1. 先读本地 `localStorage`
2. 用本地结果初始化页面，保证秒开
3. 并行请求工作区列表与远端布局
4. 在远端布局 hydrate 完成或明确失败之前，不触发首次 attention 自动弹出
5. 若远端存在记录，则用远端覆盖当前可持久化字段，并同步回本地
6. `previousAttentionMap` 和 `hasHydratedAttentionSnapshot` 仍由当前页面基于实时工作区数据自行建立，不进入远端持久化

### 本地交互写回

以下动作都触发布局同步：

- 打开 pane
- 追加 pane
- 关闭 pane
- 切换 active pane
- 手动移走 attention pane
- 由于 attention 离开而清理旧 dismiss 状态

同步顺序：

1. 先更新前端状态，保证 UI 立即响应
2. 同步写本地 `localStorage`
3. 异步 `PUT /api/workspace-view`
4. 服务端写库成功后广播 `workspace_view_updated`
5. 收到广播的所有端按服务端下发值收敛，并重新写回本地缓存

远端写失败时：

- 保留当前本地 UI 状态
- 展示非阻塞错误信息
- 下一次用户交互、刷新或 websocket 重连时继续尝试对齐远端

并发规则：

- 不做前端字段级 merge
- 服务端采用 `last-write-wins`
- 以后端最近一次成功写入并广播出的完整布局为最终状态

## attention 自动弹出规则

attention 定义继续沿用现有逻辑：

- `has_pending_approval` 一定算 attention
- `running` 不自动弹
- 非运行中时，`needs_attention / has_unseen_turns` 算 attention

自动弹出与 dismiss 规则：

1. 首次加载页面时，不对“已在 attention 中”的旧任务自动弹出
2. 某任务从非 attention 进入 attention 时，若其不在 `dismissedAttentionIds` 中，则自动追加到工作区末尾
3. 用户手动移走后，将该工作区加入 `dismissedAttentionIds`
4. 在同一轮 attention 内，无论轮询还是 websocket 刷新，都不再重复自动弹出
5. 用户若手动重新打开该任务，则立即从 `dismissedAttentionIds` 中移除
6. 如果另一端本来就开着这个 pane，不强制把它被动关闭；只禁止后续再次自动弹出
7. 任务离开 attention 后，从 `dismissedAttentionIds` 中清除
8. 下次再次进入 attention 时，可再次自动弹出

## 测试范围

至少覆盖：

- 后端空布局读取
- 后端布局写入后再次读取
- 后端布局写入后广播 `workspace_view_updated`
- 前端启动时远端布局覆盖本地旧布局
- 前端布局变更会写远端并接收广播
- A 端布局变化后，B 端实时同步
- 手动移走后，同一轮 attention 不再重复自动弹
- 手动重新打开后清除 dismiss
- 离开 attention 再进入后，重新自动弹出
- websocket 断开重连后通过 `GET /api/workspace-view` 补齐
- 远端写失败时，本地 UI 仍保持可用

## 风险与控制

- 风险：页面初始时本地与远端短暂不一致
  - 控制：采用“本地秒开 + 远端覆盖 + 首次自动弹出延后到 hydrate 完成后”的明确顺序
- 风险：dismiss 规则写错导致 running 任务再次误弹
  - 控制：复用现有 `isAttentionWorkspace` 判断并补回归测试
- 风险：多端并发写入造成短暂覆盖
  - 控制：显式采用 `last-write-wins`，并把完整布局随广播下发
- 风险：前端状态和服务端结构漂移
  - 控制：新增显式类型、API 测试和 websocket 事件测试，避免自由形态 JSON
