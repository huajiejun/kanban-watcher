# 工作区远端布局同步设计

日期：2026-03-25

## 目标

为网页版工作区增加“本地与远端同步”的布局持久化能力，使同一服务在不同端口打开时共享同一份布局状态；同时把“需要注意自动弹出”的抑制规则升级为按 attention 轮次生效，保证用户手动移走后同一轮不再弹、再次进入 attention 后重新弹出。

## 已确认需求

- 工作区布局需要在本地和远端保持同步
- 远端增加，本地也增加；本地减少，远端也减少
- 前端允许继续保留本地缓存，但远端需要成为跨端共享来源
- 需要注意任务在第一次进入 attention 时自动弹出
- 用户手动移走后，同一轮 attention 不再自动弹出
- 若任务离开 attention 后再次进入 attention，应视为新一轮 attention，可以重新自动弹出

## 现状

- 前端仅在 `localStorage` 保存 `openWorkspaceIds` 与 `activeWorkspaceId`
- attention 自动弹出的抑制状态仅存在当前浏览器内存中
- 后端尚无工作区布局读写接口
- 后端已存在 MySQL 存储层和 `/api/workspaces/active` 等工作区接口，可在同一 API 体系内扩展

## 方案对比

### 方案一：远端单一真源，本地作为镜像缓存

前端启动先读本地做快速恢复，再拉取远端布局并覆盖本地；任何布局变更都先更新当前界面，再异步写远端，远端成功后回写本地镜像。

优点：

- 跨端一致性最好
- 兼顾秒开和远端共享
- 现有本地恢复链路可以继续复用

缺点：

- 需要新增后端表和接口
- 需要处理首次远端覆盖本地的时序

### 方案二：布局远端化，attention 抑制仍留本地

只同步打开列表和活动窗格，手动移走 attention 的抑制状态仍存在本地。

优点：

- 实现更快

缺点：

- 不同端口对“是否该再次弹出”判断不一致
- 不满足用户对跨端一致交互的要求

### 方案三：服务端内存态同步

由 HTTP 服务进程在内存里保存布局，不落库。

优点：

- 改动最少

缺点：

- 服务重启即丢失
- 不是持久化，不满足远端保存目标

## 选型

采用方案一：远端单一真源，本地作为镜像缓存。

## 数据模型

后端新增一张布局表，例如 `kw_workspace_views`，保存单一共享布局记录：

- `scope_key`
- `open_workspace_ids_json`
- `active_workspace_id`
- `dismissed_attention_state_json`
- `updated_at`
- `created_at`

其中：

- `open_workspace_ids_json` 保存当前右侧 pane 顺序
- `dismissed_attention_state_json` 保存按 attention 轮次记录的手动移走信息

前端页面状态扩展为：

- `openWorkspaceIds`
- `activeWorkspaceId`
- `dismissedAttentionState`
- `previousAttentionMap`
- `hasHydratedAttentionSnapshot`

`dismissedAttentionState` 结构为：

- key：`workspaceId`
- value：当前被手动移走的 attention 轮次标识

attention 轮次标识不单独依赖服务端计数器，而是由前端在状态收敛时基于“上一次 attention 布尔值 -> 这一次 attention 布尔值”的跃迁生成并持久化；只要远端布局保存的是最新轮次，跨端刷新就能得到一致抑制行为。

## 接口设计

新增两个 HTTP API：

### `GET /api/workspace-view`

返回：

- `open_workspace_ids`
- `active_workspace_id`
- `dismissed_attention_state`

无记录时返回空布局对象，不报错。

### `PUT /api/workspace-view`

请求体：

- `open_workspace_ids`
- `active_workspace_id`
- `dismissed_attention_state`

语义：

- 以整份布局覆盖远端当前记录
- 服务端负责做基础字段校验和 JSON 持久化

## 前端同步规则

### 启动恢复

1. 先读本地 `localStorage`
2. 用本地结果初始化页面，保证秒开
3. 再请求远端布局
4. 若远端存在记录，则用远端覆盖当前可持久化字段，并同步回本地
5. `previousAttentionMap` 和 `hasHydratedAttentionSnapshot` 仍由当前页面基于实时工作区数据自行建立，不进入远端持久化

### 本地交互写回

以下动作都触发布局同步：

- 打开 pane
- 追加 pane
- 关闭 pane
- 切换 active pane
- 手动移走 attention pane
- 由于 attention 离开而清理旧 dismiss 状态

同步顺序：

1. 先更新前端状态，保证 UI 立即响应
2. 同步写本地 `localStorage`
3. 异步写远端接口
4. 若远端写成功，将服务端确认值重新写回本地

远端写失败时：

- 保留当前本地 UI 状态
- 展示非阻塞错误信息
- 下一次用户交互或刷新时继续尝试覆盖远端

## attention 自动弹出规则

attention 定义继续沿用现有逻辑：

- `has_pending_approval` 一定算 attention
- `running` 不自动弹
- 非运行中时，`needs_attention / has_unseen_turns` 算 attention

轮次规则：

1. 首次加载页面时，不对“已在 attention 中”的旧任务自动弹出
2. 某任务从非 attention 进入 attention 时，形成新一轮 attention
3. 若该工作区当前轮次未被手动移走，则自动追加到工作区末尾
4. 用户手动移走后，记录“该工作区当前轮次已 dismissed”
5. 在同一轮 attention 内，无论轮询还是 websocket 刷新，都不再重复自动弹出
6. 任务离开 attention 后，旧 dismiss 状态被清掉
7. 下次再次进入 attention，视为新一轮 attention，可再次自动弹出

## 测试范围

至少覆盖：

- 后端空布局读取
- 后端布局写入后再次读取
- 前端启动时远端布局覆盖本地旧布局
- 前端布局变更会写远端
- 手动移走后，同一轮 attention 不再重复自动弹
- 离开 attention 再进入后，重新自动弹出
- 远端写失败时，本地 UI 仍保持可用

## 风险与控制

- 风险：页面初始时本地与远端短暂不一致
  - 控制：采用“本地秒开 + 远端覆盖”的明确顺序
- 风险：dismiss 规则写错导致 running 任务再次误弹
  - 控制：复用现有 `isAttentionWorkspace` 判断并补回归测试
- 风险：前端状态和服务端结构漂移
  - 控制：新增显式类型和序列化测试，避免自由形态 JSON
