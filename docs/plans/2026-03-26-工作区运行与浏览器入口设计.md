# 工作区运行与浏览器入口设计

## 1. 背景

当前 `kanban-watcher` 的工作区主页已经支持：

- 按状态分组展示工作区
- 在桌面端打开多个工作区会话面板
- 展示 `has_running_dev_server` 等摘要状态

但它还缺少两项与 `vibe-kanban` 对齐的工作区级能力：

1. 在工作区卡片中直接启动预设的 dev server 脚本
2. 在工作区边上提供一个按钮，直接打开该工作区预设的浏览器目标页

本次需求已经明确以下边界：

- 只支持启动预设的 dev server 脚本，不扩展到 setup/cleanup/archive 等其他脚本
- “打开浏览器”打开的是工作区的预设穿透地址，而不是本地预览地址

## 2. 目标

### 2.1 功能目标

1. 每张工作区卡片提供“运行”操作，点击后启动该工作区预设的 dev server
2. 工作区主页侧栏旁提供“打开浏览器”按钮，对当前激活工作区生效
3. 当前激活工作区存在预设穿透地址时，可一键打开对应页面
4. 前端不拼装、不猜测穿透地址，只消费后端显式返回的地址

### 2.2 非目标

1. 不新增停止 dev server 按钮
2. 不支持运行 setup/cleanup/archive 等其他脚本
3. 不在后端主动打开本机 GUI 浏览器
4. 不修改 7777 端口占用处理策略；如联调冲突，只提示用户处理

## 3. 设计原则

1. 运行状态和浏览器地址都以工作区摘要数据为准，不让前端推断
2. 卡片“选中工作区”和“运行脚本”是两个不同交互，避免事件冒泡混淆
3. 接口尽量贴近 `vibe-kanban` 现有能力命名，便于后续对齐
4. 缺少穿透地址时展示禁用态和明确提示，不把它当作异常

## 4. 方案对比

### 方案 A：扩展工作区摘要并新增启动接口（采用）

- 在工作区摘要中新增 `browser_url`
- 复用已有 `has_running_dev_server`
- 新增启动 dev server 的 HTTP 接口
- 前端列表刷新后统一从摘要中获得运行态与浏览器地址

优点：

- 前后端职责清晰
- 前端不需要额外查询浏览器地址
- 与 `vibe-kanban` 的“显式接口驱动”方式最接近

缺点：

- 需要同时修改 Go 接口、类型映射和前端单测

### 方案 B：单独查询浏览器地址接口

- 工作区摘要只保留运行状态
- 侧栏按钮点击前单独请求一次浏览器地址

优点：

- 对现有摘要结构侵入较小

缺点：

- 切换激活工作区时前端需要额外请求
- 按钮态和数据态更容易短暂不一致

### 方案 C：前端按规则拼装穿透地址

优点：

- 看起来实现快

缺点：

- 强依赖外部配置约定
- 地址规则变化会直接导致功能失效
- 不符合本次“与 `vibe-kanban` 保持一致”的目标

## 5. 最终设计

### 5.1 数据模型

扩展前端工作区类型与后端工作区摘要返回，增加：

- `browser_url?: string`

保留并继续使用：

- `has_running_dev_server`

语义约束：

1. `browser_url` 表示当前工作区预设的穿透访问地址
2. `browser_url` 为空时代表当前工作区没有可打开的浏览器目标
3. `has_running_dev_server` 为 `true` 时，前端将“运行”按钮视为运行中/禁用态

### 5.2 后端接口

新增接口：

- `POST /api/workspace/{workspace_id}/dev-server`

行为：

1. 复用当前项目已有的单数前缀工作区动作体系，避免新增第二套路由风格
2. 根据工作区 ID 触发预设 dev server 启动
3. 成功时返回成功响应
4. 若已经存在运行中的 dev server，可返回成功或幂等型业务响应，由前端刷新后展示运行中状态

扩展已有接口：

- `GET /api/workspaces/active`

返回的每个 workspace 增加：

- `browser_url`
- 继续保留 `has_running_dev_server`

### 5.3 前端交互

#### 卡片内“运行”按钮

落点：

- `src/components/workspace-section-list.ts`

规则：

1. 卡片不再使用整块 `<button>` 作为根节点，改为非交互容器包裹“主操作按钮”和“运行按钮”两个 sibling 交互元素
2. 主操作按钮仍然只负责选中工作区
3. “运行”按钮位于卡片操作区，点击时不触发选中逻辑
4. 若工作区 `has_running_dev_server=true`，按钮展示“运行中”禁用态
5. 若当前工作区正在发起启动请求，按钮展示 loading/禁用态
6. 请求成功后刷新工作区列表，等待摘要态回流

#### 侧栏旁“打开浏览器”按钮

落点：

- `src/web/workspace-home.ts`

规则：

1. 按钮绑定当前 `activeWorkspaceId`
2. 当前工作区存在 `browser_url` 时可点击
3. 点击后执行 `window.open(browserUrl, "_blank", "noopener")`
4. 当前工作区无 `browser_url` 时按钮禁用，并展示固定提示

### 5.4 错误处理

1. 启动 dev server 失败时，在工作区页面局部反馈区展示错误文案
2. `browser_url` 为空不视为接口失败，只显示不可用态
3. 如果后端返回“已在运行”，前端不报致命错误，直接刷新并同步状态

## 6. 代码改动范围

### 前端

- `src/types.ts`
  - 扩展工作区类型字段
- `src/lib/http-api.ts`
  - 新增启动 dev server 请求方法
- `src/components/workspace-section-list.ts`
  - 将卡片根结构改为非按钮容器
  - 新增主操作按钮、卡片操作区和“运行”按钮事件
- `src/web/workspace-home.ts`
  - 管理启动请求状态
  - 新增“打开浏览器”按钮
  - 处理 `window.open` 与局部反馈

### 后端

- `internal/api/messages.go`
  - 活跃工作区响应增加 `browser_url`
- `internal/api/types.go`
  - 扩展工作区摘要类型
- `internal/server/server.go`
  - 扩展单数前缀工作区动作路由，接入 `POST /api/workspace/{workspace_id}/dev-server`
- 与 dev server 启动相关的服务实现文件
  - 新增或复用启动逻辑，供工作区动作路由调用

如果现有后端已有可复用的启动与穿透地址获取服务，应优先复用，不重复造新的业务层。

## 7. 测试策略

### 前端测试

1. `workspace-section-list`：
   - 渲染“运行”按钮
   - 主操作按钮与运行按钮为 sibling 结构，不出现嵌套交互元素
   - 运行按钮点击不会触发 `workspace-select`
   - 运行中时按钮禁用
   - 重复点击启动时会防重入
2. `workspace-home`：
   - 激活工作区存在 `browser_url` 时，“打开浏览器”按钮可用
   - 无 `browser_url` 时按钮禁用
   - 无 `activeWorkspaceId` 时按钮禁用
   - 点击按钮调用 `window.open`
   - 启动失败时展示局部错误反馈
   - 启动 dev server 后触发工作区列表刷新

### 后端测试

1. 活跃工作区接口返回 `browser_url`
2. `POST /api/workspace/{id}/dev-server` 的 method/path 合约测试
3. 启动 dev server 接口成功与幂等场景
4. 缺少浏览器地址时接口返回空值而不是错误

## 8. 风险与控制

1. 风险：后端暂时没有稳定的穿透地址来源
   - 控制：先在后端封装单一 `browser_url` 输出点，前端不耦合来源
2. 风险：卡片按钮增加后，现有点击区域行为被破坏
   - 控制：前端测试覆盖冒泡与选中逻辑
3. 风险：启动接口返回成功但摘要状态回流有延迟
   - 控制：前端保留短暂 loading 态，并在成功后主动刷新列表

## 9. 验收标准

1. 用户可以从工作区卡片直接启动预设 dev server
2. 已运行的工作区不会重复触发启动
3. 选中工作区后，侧栏旁可通过“打开浏览器”按钮打开其预设穿透地址
4. 没有穿透地址的工作区不会报错，只显示禁用态
5. 前后端测试覆盖新增关键分支
