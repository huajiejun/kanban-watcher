# 发送消息中介控制方案

## 1. 背景

当前 `kanban-watcher` 已经具备以下能力：

- 从 `vibe-kanban` 同步工作区、会话、消息到本地数据库
- 为 Home Assistant 提供工作区和消息查询接口
- 通过 `POST /api/workspace/{id}/follow-up` 将消息转发给 `vibe-kanban`

但现有发送链路仍有明显耦合：

- Home Assistant 仍然依赖 `vibe-kanban` 的 follow-up 语义
- 发送消息时依赖运行时拼装参数，缺少本地持久化的执行上下文
- 暂不支持“立即发送”和“加入队列”两种统一控制模式
- 后续若 `vibe-kanban` 的接口字段变化，Home Assistant 将被迫跟随调整

本方案的目标是将 `kanban-watcher` 升级为消息控制中介层，让 Home Assistant 只与 `watcher` 交互，由 `watcher` 负责查库、补全参数、选择实际调用的 `vibe-kanban` 接口。

## 2. 目标

### 2.1 业务目标

- Home Assistant 只发送简单消息，不再关心执行器、模型、变体等细节
- `kanban-watcher` 根据本地数据库中保存的执行上下文，补齐调用 `vibe-kanban` 所需参数
- 提供统一的消息发送入口，同时支持：
  - 立即发送
  - 加入队列
- 在执行上下文缺失时提供兜底补拉逻辑，避免首轮发送直接失败

### 2.2 非目标

- 本期不支持多条排队消息管理
- 本期不支持在 Home Assistant 侧动态覆盖模型或执行器参数
- 本期不实现复杂优先级调度、自动重试、消息模板

## 3. 推荐方案

推荐采用“命令代理层 + 执行上下文仓库”方案。

核心思想：

1. `kanban-watcher` 在同步阶段保存 workspace/session 的发送上下文
2. Home Assistant 发送消息时只传：
   - `workspace_id`
   - `message`
   - `mode`
3. `kanban-watcher` 从数据库中查询该 workspace 的最新可用发送参数
4. `kanban-watcher` 根据 `mode` 调用：
   - `vibe-kanban /api/sessions/{session_id}/follow-up`
   - 或 `vibe-kanban /api/sessions/{session_id}/queue`

这样可以让 `watcher` 成为唯一稳定边界，屏蔽第三方程序接口细节。

## 4. 数据模型设计

### 4.1 新增表

建议新增一张独立表保存消息发送上下文，避免将发送配置散落在现有 `kw_workspaces` 表中。

建议表名：`kw_msg_contexts`

命名说明：

- `kw`：项目前缀
- `msg`：消息域
- `contexts`：表示某个工作区当前的发送上下文

### 4.2 表结构建议

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `workspace_id` | `VARCHAR(36)` PK | 工作区 ID |
| `session_id` | `VARCHAR(36)` NOT NULL | 当前默认发送使用的 session |
| `process_id` | `VARCHAR(36)` NULL | 最近一次关联 process，可用于调试和回溯 |
| `executor` | `VARCHAR(50)` NULL | 执行器标识 |
| `variant` | `VARCHAR(50)` NULL | 模型或变体标识 |
| `executor_config_json` | `JSON` NOT NULL | 完整执行配置，作为 follow-up/queue 的主要参数来源 |
| `force_when_dirty` | `BOOLEAN` NULL | 脏工作区策略 |
| `perform_git_reset` | `BOOLEAN` NULL | 是否允许 git reset |
| `default_send_mode` | `VARCHAR(20)` NOT NULL DEFAULT `'send'` | 默认发送模式 |
| `source` | `VARCHAR(30)` NOT NULL | 来源，例如 `sync` / `fallback_fetch` |
| `updated_at` | `TIMESTAMP(3)` NOT NULL | 业务更新时间 |
| `synced_at` | `TIMESTAMP(3)` NOT NULL | 入库时间 |

索引建议：

- 主键：`workspace_id`
- 普通索引：`session_id`
- 普通索引：`updated_at`

### 4.3 Go 模型建议

新增模型 `MessageContext`：

- `WorkspaceID`
- `SessionID`
- `ProcessID`
- `Executor`
- `Variant`
- `ExecutorConfigJSON`
- `ForceWhenDirty`
- `PerformGitReset`
- `DefaultSendMode`
- `Source`
- `UpdatedAt`
- `SyncedAt`

## 5. 数据来源与同步策略

前提说明：

- 当前仓库的同步客户端 `internal/api/client.go` 仅封装了 `/api/workspaces` 与 `/api/workspaces/summaries`
- 当前同步链路 `internal/sync/sync.go` 仅将 workspace、session、execution process、messages 写入本地数据库
- 当前仓库尚未存在稳定获取 `executor_config`、`variant` 的现成同步代码路径

### 5.1 主路径：同步阶段写库

在 `kanban-watcher` 现有同步链路中，当获取到 workspace/session/process 的远端数据后，同步提取发送上下文并写入 `kw_msg_contexts`。

写入触发点建议：

- 同步到最新 session 时
- 同步到最新 execution process 时
- 若能从远端接口直接拿到 executor config，则在此阶段直接保存
- 若第一阶段无法稳定获取 `executor_config`，则允许过渡实现：
  - 先落库 `session_id`
  - `executor_config_json` 暂时为空或保存默认配置
  - 发送阶段继续允许兜底补拉或沿用默认参数

### 5.2 兜底路径：发送前补拉

若 Home Assistant 发送消息时发现以下任一条件不满足，则触发补拉：

- `workspace_id` 对应记录不存在
- `session_id` 为空
- `executor_config_json` 为空

补拉策略：

1. 通过 `workspace_id` 向 `vibe-kanban` 查询最新可用 session
2. 查询发送该 session 所需的执行参数
3. 写回 `kw_msg_contexts`
4. 再执行消息发送

这样可避免同步延迟或老数据导致的发送失败。

### 5.3 一致性原则

- 数据库中的发送上下文是默认来源
- 远端接口是兜底校准来源
- 发送时不依赖 Home Assistant 传入执行参数

## 6. 接口设计

### 6.1 对 Home Assistant 暴露的统一接口

新增统一接口：

`POST /api/workspace/{workspace_id}/message`

请求体：

```json
{
  "message": "请继续处理这个问题",
  "mode": "send"
}
```

字段说明：

- `message`：必填，用户消息
- `mode`：可选，取值：
  - `send`：立即发送
  - `queue`：加入队列
  - 若为空，则默认使用库中的 `default_send_mode`

建议响应：

```json
{
  "success": true,
  "workspace_id": "ws_123",
  "session_id": "sess_456",
  "action": "queued",
  "message": "Message queued successfully"
}
```

### 6.2 watcher 内部调用规则

当 `mode=send`：

- 调用 `vibe-kanban /api/sessions/{session_id}/follow-up`

请求体由 `kw_msg_contexts` 补齐，例如：

```json
{
  "prompt": "请继续处理这个问题",
  "executor_config": {
    "...": "..."
  },
  "force_when_dirty": false,
  "perform_git_reset": false
}
```

当 `mode=queue`：

- 调用 `vibe-kanban /api/sessions/{session_id}/queue`

请求体示例：

```json
{
  "message": "当前任务完成后补充测试",
  "executor_config": {
    "...": "..."
  }
}
```

### 6.3 兼容旧接口

现有 `POST /api/workspace/{id}/follow-up` 可暂时保留，作为兼容入口。

兼容策略建议：

- 内部转调统一发送服务
- 默认 `mode=send`
- 后续文档中标记为兼容接口，不再推荐 Home Assistant 新接入使用

## 7. 发送流程设计

### 7.1 标准流程

1. Home Assistant 调用 `POST /api/workspace/{id}/message`
2. `watcher` 校验 API Key、请求体和 `mode`
3. `watcher` 查询 `kw_msg_contexts`
4. 若上下文完整，则直接组装远端请求
5. 若上下文缺失，则执行补拉并回写数据库
6. 根据 `mode` 调用 `follow-up` 或 `queue`
7. 返回统一结果给 Home Assistant

### 7.2 错误处理

以下情况应明确区分：

- 工作区不存在：返回 `404`
- 找不到可用 session：返回 `409` 或 `404`
- 本地上下文缺失且补拉失败：返回 `502`
- 远端 follow-up/queue 失败：返回 `502`
- 请求参数非法：返回 `400`

返回体建议包含：

- `success`
- `error`
- `workspace_id`
- `session_id`（若可用）

## 8. 代码结构建议

### 8.1 Store 层

新增：

- `MessageContext` 模型
- schema 初始化语句
- `UpsertMessageContext`
- `GetMessageContextByWorkspaceID`

可选新增：

- `DeleteMessageContext`
- `ListStaleMessageContexts`

### 8.2 API 代理层

现有 `ProxyClient` 建议扩展为：

- `SendFollowUpWithContext`
- `QueueMessageWithContext`
- `FetchMessageContext`

避免继续将“获取 session + 组请求 + 调远端”写死在单个 follow-up 函数里。

### 8.3 服务层

建议新增独立消息发送服务，例如：

- `internal/service/message_dispatcher.go`

职责：

- 查询消息上下文
- 必要时补拉
- 根据 mode 选择 follow-up 或 queue
- 统一错误映射

这样 HTTP handler 只负责解析请求和返回响应，不承担业务编排。

### 8.4 HTTP 层

新增 handler：

- `handleWorkspaceMessage`

保留兼容 handler：

- `handleFollowUp`

建议让兼容 handler 内部直接调用统一服务，并固定 `mode=send`。

## 9. 分阶段实施建议

### 前置调研阶段

- 确认 `vibe-kanban` 是否存在稳定可调用的“获取 session 执行配置”接口
- 若没有稳定接口，确认第一版是否接受沿用当前默认执行配置的过渡方案
- 确认 Home Assistant 卡片将从旧 `/follow-up` 接口切换到统一 `/message` 接口

### 第一阶段

- 新增 `kw_msg_contexts` 表
- 同步链路支持写入消息上下文
- 新增统一接口 `/api/workspace/{id}/message`
- 支持 `mode=send|queue`
- 发送时从数据库补全 `session_id` 和 `executor_config`
- 保留旧 follow-up 接口兼容
- 前端卡片接入真实 queue/send 接口，移除“仅界面展示”的假队列逻辑

### 第二阶段

- 增加上下文缺失时的补拉能力
- 增加 queue 状态查询或取消接口的 watcher 封装
- 为 Home Assistant 返回更明确的排队状态说明

### 第三阶段

- 支持 `mode=auto`
- 支持更细粒度的默认发送策略
- 支持审计日志、发送历史和重试策略

## 11. 风险与注意事项

- 当前仓库尚未具备稳定获取 `executor_config` 的数据源：`internal/api/client.go` 只封装了 workspace/summaries 查询，而 `internal/api/proxy.go` 仍写死默认配置。因此“任务创建时完整保存模型/执行器配置”在实现前需要先补齐远端读取路径，或接受首版过渡方案
- `latest_session_id` 与真实可发送 session 可能短暂不一致，必须允许发送前补拉
- queue 接口语义与 follow-up 不同，响应结构也不同，watcher 需要统一包装
- 不应让 Home Assistant 直接传入未受控的执行参数，否则会破坏“中介层稳定边界”
- 当前前端卡片的 queue 仅是本地 UI 状态，不是后端真实排队。接入该方案时，必须同步修改 `src/lib/http-api.ts` 与 `src/kanban-watcher-card.ts`，避免出现“界面显示已排队，但后端并未入队”的误导

## 12. 结论

本方案将 `kanban-watcher` 从“消息转发器”提升为“消息控制中介层”：

- 对外提供稳定、简单的 Home Assistant 接口
- 对内持久化 `vibe-kanban` 所需执行上下文
- 通过数据库查询补齐调用参数
- 支持立即发送和加入队列两种模式

这是当前需求下复杂度和可维护性最平衡的方案，适合作为后续实现的第一版基线。
