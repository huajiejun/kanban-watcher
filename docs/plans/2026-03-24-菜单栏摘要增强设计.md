# 2026-03-24 菜单栏摘要增强设计

## 目标

在 macOS 菜单栏下拉中，为需要关注或最近有动态的工作区补充一条单行摘要，让用户在不打开主界面的前提下快速理解“为什么要关注这个任务”。

## 设计范围

- 菜单栏下拉从“仅任务名”升级为“任务名 + 单行摘要”
- 摘要优先展示原因提示，其次展示最近一条用户可读消息
- 只增强菜单栏信息密度，不在本阶段加入点击操作、筛选或二级详情面板

## 非目标

- 不在菜单栏内展示完整消息列表
- 不新增菜单栏内交互动作
- 不让托盘层自己访问数据库或额外发起网络请求

## 菜单结构

### 展示形态

每个工作区保持一行主标题，并按需增加一行副文本：

- 第一行：`状态符号 + 工作区名称`
- 第二行：`最近一条消息摘要` 或 `需要关注的原因提示`

### 展示规则

- 只有“需要关注”或“最近有新消息”的工作区显示第二行摘要
- 普通空闲工作区仍只显示一行
- 第二行最多显示一行，超长截断
- 当前无法生成摘要时，回退为单行显示

## 摘要来源与优先级

### 优先级

1. 原因提示
2. 最近一条用户可读消息
3. 回退提示

### 原因提示示例

- `待审批：等待你确认下一步`
- `未读消息：对方刚回复了部署结果`
- `运行失败：测试脚本退出码 1`
- `同步失败：最近一次同步未完成`

### 最近消息提取规则

- 只保留对人有意义的自然语言文本
- 排除工具日志、纯代码块、空文本、无意义占位文本
- 将多行文本压缩为单行
- 移除 markdown 标记和明显噪声

### 回退提示

- `最近暂无可读摘要`
- `运行中，等待下一步输出`
- `已完成，暂无新动态`

## 数据流与职责边界

### 推荐边界

- 上游数据模型负责提供菜单摘要候选
- 托盘层只负责显示裁剪和回退策略

### 具体职责

#### `api.EnrichedWorkspace`

- 增加菜单摘要候选字段，或补齐构建摘要所需的原始字段
- 提供“原因提示”和“最近消息摘要”的上游输入

#### `tray.App`

- 判断当前工作区是否需要显示第二行
- 执行截断与单行清洗
- 将主标题和副文本组装为菜单项

### 约束

- 托盘层不直接查数据库
- 托盘层不直接请求 HTTP API
- 摘要规则应尽量复用，避免菜单栏和其他展示端各自实现一套

## 文本清洗与截断

### 清洗规则

- 去除换行
- 去除 markdown 标记
- 去除代码块围栏
- 折叠连续空白字符

### 截断规则

- 单行展示
- 建议保留 32 个可见字符左右
- 超出部分追加 `...`

## 错误兜底

- 摘要为空或解析失败：回退到原因提示
- 原因提示也缺失：回退到单行任务名
- 单个工作区摘要异常：只影响该项，不影响整个菜单刷新
- 所有摘要都无法生成：菜单保持当前版本可用性

## 测试方案

### 单元测试

- 原因提示优先于最近消息
- 最近消息为空时能回退
- markdown 清洗与单行压缩正确
- 长文本截断规则正确

### 托盘层测试

- 有摘要时生成两行显示
- 无摘要时保持单行显示
- 普通任务不会误显示副文本

### 人工验证

- 至少验证一个待审批任务显示原因提示
- 至少验证一个有最近消息的任务显示摘要
- 至少验证一个普通任务保持简洁显示

## 实施顺序

1. 在上游数据模型中补菜单摘要候选
2. 在托盘层实现两行显示和回退逻辑
3. 补清洗、截断和测试
4. 做菜单栏人工验证并保留关键截图

## 风险与取舍

- 菜单栏空间有限，摘要必须严格截断，否则很容易撑高菜单
- 如果摘要完全在托盘层拼接，后续维护会分散；因此优先推荐上游生成候选摘要
- 第一版应优先保证稳定和可读，不追求在菜单里展示过多字段
