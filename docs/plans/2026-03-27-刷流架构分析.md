# 刷流架构分析

> 日期：2026-03-27
> 范围：`vibe-kanban` 前端消息去重 + `kanban-watcher` 当前刷流架构

## 1. 结论摘要

当前这套“刷流”实际上分成两层：

1. `vibe-kanban` 前端负责消费 `normalized-logs/ws` 的 JSON Patch 流，并在 **patch 批次级** 做去重。
2. `kanban-watcher` 前端负责消费 watcher 后端广播的 `session_messages_appended`，并在 **消息对象级** 做幂等合并。

这意味着：

1. `vibe-kanban` 的核心去重单位不是“聊天气泡”，而是 **同一路径的 JSON Patch 操作**。
2. `kanban-watcher` 的核心去重单位才是“已归一化后的会话消息”。
3. watcher 的增量消息来源并不是直接照搬 `vibe-kanban` 前端实现，而是先经 watcher 后端落库、再广播给前端。

## 2. vibe-kanban 前端如何对重复消息去重

### 2.1 主链路

`vibe-kanban` 聊天历史的实时数据流在 `useConversationHistory` 中接入：

- 运行中的 process 通过 `streamJsonPatchEntries(url, ...)` 订阅 `normalized-logs/ws`
- 每次收到一批 JSON Patch 后，先在内存里合并，再映射成带 `patchKey` 的 entries 给 UI

对应代码位置：

- `packages/web-core/src/features/workspace-chat/model/hooks/useConversationHistory.ts`
- `packages/web-core/src/shared/lib/streamJsonPatchEntries.ts`

### 2.2 真正的去重点

`streamJsonPatchEntries` 会把同一帧内收到的 patch 暂存到 `pendingOps`，到下一帧 `flush()` 时执行：

1. 调用 `dedupeOps(ops)`
2. 用 `path` 作为键记录最后一次操作
3. 仅保留每个 `path` 的最后一条 patch
4. 再通过 `applyUpsertPatch` 应用到本地 `snapshot.entries`

所以它的去重规则是：

- **同一批次内，针对同一路径的重复 add/replace，只保留最后一次写入**
- 这是“最后写 wins”的 patch 去重，不是按文本内容比较

示例：

- 同一批里先来 `add /entries/4`
- 又来 `replace /entries/4`
- 最终只保留后者

### 2.3 这层去重解决了什么

它主要解决的是流式 websocket/patch 场景里的两类重复：

1. 同一批消息里，对同一 entry 索引的重复覆盖
2. 某个 entry 先 add 后 replace 的中间态抖动

它**不负责**按“内容是否一样”去重，也**不负责**跨批次、跨重连的业务语义去重。它依赖的是：

- patch 路径稳定
- `entries[index]` 代表同一条 normalized entry

## 3. kanban-watcher 当前架构里的去重点

### 3.1 后端广播层

watcher 后端把实时事件统一收口到 `/api/realtime/ws`：

- `workspace_snapshot`：工作区快照广播
- `session_messages_appended`：指定 session 的增量消息广播
- `workspace_view_updated`：布局同步广播

`session_messages_appended` 只发给订阅了对应 `session_id` 的客户端，不做内容级去重；它默认相信数据库写入链路已经是幂等的。

### 3.2 前端 append 层

前端在 `workspace-home.ts` 中处理 `session_messages_appended`：

1. 找到当前 `session_id` 对应的 workspace
2. 先经过 `shouldAcceptRealtimeMessages()` 做“刷流门控”
3. 把当前对话区消息拍平成一维列表
4. 用消息稳定键建 `indexByKey`
5. 新消息到来后：
   - key 已存在：覆盖旧消息
   - key 不存在：追加新消息
6. 最后按时间排序，再重新分组 tool message

### 3.3 watcher 的消息稳定键

`src/lib/dialog-messages.ts` 中 `buildDialogMessageKey()` 的优先级：

1. `process_id + entry_index`
2. `id`
3. `timestamp + content`

所以 watcher 前端的真正消息去重策略是：

- **优先用 process 维度的 entry 序号做幂等主键**
- 其次才退化到数据库 ID
- 最后才退化到弱键 `timestamp + content`

这比纯内容比较稳定得多，因为它和 watcher 后端的消息模型是一致的。

## 4. 当前刷流架构梳理

### 4.1 源头：vibe-kanban

根据现有设计文档和源码约束，真实数据流是三段式：

1. 工作区流：发现 workspace，更新 `latest_session_id`
2. session 进程流：发现 session 下有哪些 execution process
3. process 消息流：按 `process_id` 订阅 `normalized-logs/ws`，这才是真正的对话消息源

### 4.2 watcher 后端

watcher 的职责不是简单透传，而是中介层：

1. 从 vibe-kanban 拉工作区/会话/process/entry
2. 本地落库
3. 对外提供 HTTP 查询接口
4. 当新增消息写库成功后，再广播 `session_messages_appended`

因此：

- HTTP 是权威快照通道
- WebSocket 是增量通道
- 真正的防重最终还是依赖本地结构化主键，而不是 UI 文本比较

### 4.3 watcher 前端

前端页面分两条实时链：

1. 板级 websocket：收 `workspace_snapshot`，更新工作区列表
2. 会话级 websocket：只对当前活动 pane 的 `session_id` 建连，收 `session_messages_appended`

其中会话级链路还有一个额外门控：

- `running` 工作区持续接收增量
- `idle/attention` 且最后一条消息已经终态时，忽略后续 append
- 避免卡片结束后仍继续“刷流”

## 5. 架构判断

### 5.1 优点

1. 职责拆分清楚：`vibe-kanban` 处理 patch 级去重，watcher 处理消息级幂等。
2. 键设计稳定：`process_id + entry_index` 是整条链里最可靠的消息身份。
3. HTTP + WebSocket 双通道合理：断线可以靠 HTTP 补拉恢复。
4. watcher 前端增加了“终态门控”，能减少无意义滚动和重复刷新。

### 5.2 风险点

1. 如果上游某些消息缺失 `process_id` / `entry_index`，前端会退化到弱键，重复判断就会变脆。
2. watcher 前端当前只对活跃 pane 建 session websocket，非活跃 pane 依赖轮询补偿，实时性较弱但实现更稳。
3. `vibe-kanban` 的 patch 去重是批次内按 `path` 去重，不等于业务语义去重；如果上游跨批次重复发送同一 entry，仍需依赖更下游的结构化键。

## 6. 最关键的理解

这套架构里，“消息去重”不是单点完成，而是分层完成：

1. `vibe-kanban`：先把 websocket patch 流压平，避免同一批次重复改同一个 entry
2. watcher 后端：按结构化主键落库，避免重复写入
3. watcher 前端：按稳定键合并 append，避免 UI 重复追加

所以如果你要分析“刷流架构”，最准确的说法不是“前端做了去重”，而是：

**上游 patch 去抖 + 中间层主键幂等 + 前端 append 合并** 共同构成了当前的防重链。
