# 对话消息 Redis 优先查询设计

**日期**: 2026-03-29
**状态**: 已确认

## 目标

优化工作区对话消息加载性能：从 Redis 直接读取最新消息（尚未 flush 到 MySQL 的数据也可立即可见），默认显示最近 50 条，支持上划 cursor 分页加载更多，跨多个 processID 汇总。

## 当前架构

```
前端点击工作区 → /api/workspaces/{id}/latest-messages
  → MySQL process_entries 表（全量加载）
  → 无分页，一次性返回所有消息
```

痛点：Redis buffer 中的最新数据需等 flush 才可见；无分页机制。

## 关系模型

```
Workspace 1:N Session 1:N ExecutionProcess 1:N ProcessEntry
```

当前 `latest-messages` 只查 workspace 的 `latest_session_id`。
新设计同样只查 `latest_session_id` 下的所有 processID（覆盖绝大多数场景），
历史 session 通过切换 session 查看。

## 方案：Redis 优先 + MySQL 兜底

### Redis 数据结构变更

新增 workspace 级 process 索引：

```
workspace_processes:{workspaceID}  →  ZSET
  member: processID
  score:  最新消息的 entry_timestamp UnixMilli
```

现有 Key 不变：
- `process_entries:{processID}` → ZSET（score=entry_index）
- `process_entry_data:{processID}` → Hash
- `process_last_idx:{processID}` → String

### 写入侧

在 `RedisBuffer.Enqueue` 的 Pipeline 中追加一步 ZADD：

```go
wsZsetKey := "workspace_processes:" + entry.WorkspaceID
pipe.ZAdd(ctx, wsZsetKey, &redis.Z{
    Score:  float64(entry.EntryTimestamp.UnixMilli()),
    Member: processID,
})
pipe.Expire(ctx, wsZsetKey, 7*24*time.Hour)  // 7 天，比消息 TTL 更长
```

ZADD 对同一 member 幂等，score 更新为最新消息时间。

### Cursor 格式

采用三段式组合 cursor，确保跨 processID 唯一性：

```
{entry_timestamp_milli}:{processID}:{entry_index}
```

示例：`1743244800000:proc-abc-123:42`

- `entry_timestamp_milli`：消息时间戳毫秒，用于按时间定位
- `processID`：避免同时间戳跨 process 冲突
- `entry_index`：同一 process 内的序号

Cursor 通过 base64 编码后传递，前端不需要理解内部结构。

### 查询 API

新增端点：

```
GET /api/workspaces/{id}/messages?limit=50&before={cursor}
```

参数：
- `limit`：默认 50，最大 200
- `before`：cursor（base64 编码的三段式），不传则取最新

查询流程：

1. **获取 processID 列表**
   - `ZREVRANGE workspace_processes:{workspaceID} 0 -1` → 按 score 倒序
   - 如果 workspace_processes 不存在（Redis 不可用或 key 过期），降级到 MySQL

2. **从 Redis 读取**（逐个 process，倒序遍历）
   - 如果有 cursor，解析出 `minTimestamp` 和 `minEntryIndex`
   - 对每个 processID：
     - `ZREVRANGEBYSCORE process_entries:{processID} +inf {minTimestamp} LIMIT 0 {remaining}`
     - `HMGET process_entry_data:{processID} {members}`
     - 反序列化为 ProcessEntry

3. **合并排序**
   - 所有 process 的结果按 `entry_timestamp` 倒序排列
   - 取前 limit 条

4. **MySQL 兜底**
   - Redis 数据不足 limit 时，以 Redis 最旧消息的 timestamp 作为 `before`
   - 调用 `dbStore.GetSessionMessages(ctx, sessionID, remaining, before, types)` 从 MySQL 补充
   - 只查 `latest_session_id` 下的消息

5. **去重**
   - 策略：Redis 的最旧消息 timestamp > MySQL 查询的 before，天然无重叠
   - 额外保障：用 `(processID, entryIndex)` map 去重

6. **返回**
   ```json
   {
     "messages": [...],
     "has_more": true,
     "cursor": "base64(1743244800000:proc-abc:42)"
   }
   ```
   cursor 取最后一条消息的三段式值。

### Redis 降级策略

Redis 不可用时，直接查 MySQL（等同现有行为）：

```go
if rdb == nil {
    // Redis 不可用，直接走 MySQL（等同现有 latest-messages 逻辑）
    getSessionMessagesInternal(w, r, dbStore, sessionID, workspaceName, false)
    return
}
```

### 前端集成

**改动文件**：
- `src/lib/http-api.ts`：新增 `fetchWorkspaceMessages({workspaceId, limit, before})`
- `src/web/workspace-home.ts`：
  - `loadWorkspaceMessages` 改为调用新 API（默认 limit=50）
  - 新增 `loadMoreMessages(workspaceId)` 处理滚动到顶部加载更多
  - 新增 `cursorByWorkspace: Record<string, string>` 和 `hasMoreByWorkspace: Record<string, boolean>`
- `src/components/workspace-conversation-pane.ts`：滚动到顶部时触发 `loadMore`

消息列表排序：最新消息在最下方（聊天界面标准布局），滚动到顶部加载更早的消息。

**不改的部分**：
- 现有 `/api/workspaces/{id}/latest-messages` API 保持不动
- Realtime WebSocket 推送逻辑不变
- 消息渲染组件（DialogMessage）不变

### 数据流

```
首次加载（进入工作区）：
  GET /api/workspaces/{id}/messages?limit=50
  → Redis 读最新 → 不够则 MySQL 补充 → 返回 50 条 + cursor

上划加载更多（滚动到顶部）：
  GET /api/workspaces/{id}/messages?limit=50&before={cursor}
  → Redis + MySQL 查更早消息 → 返回 50 条 + 新 cursor

Realtime 新消息：
  WebSocket 推送 append 到列表底部（不变）
```
