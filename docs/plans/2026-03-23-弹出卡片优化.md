# 弹出卡片优化设计

日期：2026-03-23

## 目标

优化工作区弹窗中的消息展示，新增工具调用摘要卡片，并重点提升手机端的可读性与操作效率。

本次优化目标：

- 工具调用直接插入现有消息流，不额外拆分独立区域
- 工具调用默认只展示一行摘要，展开后再查看详情
- 工具调用视觉层级弱于正常对话消息
- 图标语义参考 `vibe-kanban`，按工具类型稳定映射
- 手机端优先保证快速扫读，不让工具调用占满整屏

## 当前现状

当前弹窗只渲染两类内容：

- 用户消息
- AI 消息

工具调用、命令执行、文件读取、搜索等中间过程没有结构化展示，导致用户只能看到最终回复，无法快速判断代理过程和当前状态。

## 设计结论

采用“同一消息流混排”的方式渲染工具调用，不增加新的面板分区。

消息流内允许出现三类消息项：

- 用户消息
- AI 消息
- 工具调用摘要卡片

工具调用卡片作为弱化信息层出现，默认单行，支持点击展开详情。

## 信息结构

### 默认态

工具调用卡片默认单行展示以下内容：

- 左侧：工具类型图标
- 中部：工具名称或动作名称
- 中部补充：极短摘要
- 右侧：状态标记

显示顺序示例：

`📄 读取文件 src/styles.ts 成功`

`> 执行命令 npm test 运行中`

默认态要求：

- 严格单行显示
- 超出内容省略
- 不显示多行正文
- 不显示长输出

### 展开态

点击工具调用卡片后展示详情区域，详情允许多行：

- 完整标题
- 命令、路径、URL 等关键上下文
- 输出摘要或结构化结果
- 必要时展示代码块或 JSON 文本

展开态只影响当前卡片，不改变整体消息流结构。

## 视觉与状态规则

### 层级

工具调用卡片视觉层级低于普通消息：

- 背景更淡
- 边框更轻
- 文本颜色更弱
- 不使用大面积高饱和色块

正常用户消息和 AI 消息仍然保持主视觉地位。

### 状态强调

仅以下状态允许明显强调：

- `pending_approval`
- `error`

其余状态保持弱化处理：

- `running`
- `success`
- `denied`

### 状态表现

- `running`：弱化底色 + 小型加载指示
- `success`：低饱和成功色
- `pending_approval`：橙色强调
- `error`：红色强调
- `denied`：灰红弱化

## 图标规则

图标语义参考 `vibe-kanban` 的工具类型映射方式，在本项目中优先使用轻量、稳定、无需新依赖的符号图标或文本图标。

推荐映射：

- `file_read`：文件图标
- `search`：放大镜图标
- `web_fetch`：地球或链接图标
- `command_run`：终端图标
- `task_create`：任务或分支图标
- `todo_management`：清单图标
- 未知工具：通用工具图标

图标职责：

- 帮助用户在手机端快速扫读
- 用最少文字区分调用类型
- 保持同一工具类型的视觉稳定性

## 手机端优化原则

手机端是这次优化的重点场景。

### 默认态要求

- 所有工具调用保持单行
- 左图标、标题、状态尽量在一行内保留
- 摘要优先截断，不让状态标签被挤掉
- 不做左右分栏

### 展开态要求

- 详情区域与主卡片区分层级
- 代码块和长文本允许横向滚动
- 展开后的边距和圆角保持紧凑，避免一张卡占据过高屏幕空间

### 消息流要求

- 工具调用与消息混排，不打断上下文
- 连续多次工具调用时仍然能快速区分用户消息和工具过程
- 工具卡片不能抢占输入区与消息主内容的空间感

## 实现边界

本次只做以下范围：

- 在弹窗消息列表中新增工具调用摘要卡片渲染
- 为工具调用增加图标、状态、默认单行摘要和展开详情
- 补齐手机端样式规则

本次不做：

- 独立工具调用面板
- 复杂的工具调用聚合
- 新图标库依赖接入
- 会话模型大规模重构

## 验证重点

需要覆盖以下验证场景：

- 用户消息、AI 消息、工具调用混排
- 多个连续工具调用的单行展示效果
- 长摘要在桌面端和手机端的截断效果
- `running`、`success`、`pending_approval`、`error` 的状态表现
- 点击展开后的详情展示
- 手机端展开后代码块或长文本的可读性

## 计划衔接

下一步实施计划应重点覆盖：

- 梳理消息数据模型，支持区分消息类型
- 为工具调用建立图标与摘要映射
- 更新消息列表渲染逻辑
- 更新弹窗和移动端样式
- 增补测试与验证文档
