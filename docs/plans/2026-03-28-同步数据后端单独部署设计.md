# 同步数据后端单独部署设计

## 1. 背景

当前 `kanban-watcher` 支持多端口实例并行运行，但只要实例连接数据库，就会启动以下能力：

- 同步服务 `sync.NewSyncService(...)`
- 远端 websocket 订阅
- 本地实时推送 `/api/realtime/ws`
- 通知链路

这导致多个业务后端如果同时部署并连接同一数据库，会重复拉取远端数据并重复入库，造成数据库和远端接口的额外压力。现有代码仅在通知环节通过 `7778` 端口做了一层限制，但同步和 websocket 仍然会在非主端口实例上启动，无法满足“只有主后端拥有实时能力”的部署目标。

## 2. 目标

- 只有主后端实例负责同步数据、消费远端 websocket、发布本地 websocket。
- 业务后端实例继续连接同一个数据库，并保留现有 HTTP API 能力。
- 前端无论从哪个业务后端访问，websocket 都统一连接主后端。
- 不拆分二进制产物，采用同一套二进制加配置/启动参数切角色。
- 尽量兼容现有默认行为，减少现网改造成本。

## 3. 非目标

- 本次不拆成两套 Go build tag 产物。
- 本次不改数据库表结构。
- 本次不重构消息分发、工作区查询等既有 HTTP 业务逻辑。
- 本次不处理“多个实例都被配置成主后端”的部署治理问题，只提供显式日志和文档约束。

## 4. 方案选型

### 4.1 方案 A：同一套二进制 + 运行时角色切换

通过新增运行角色配置，例如 `runtime.role: main | worker`，在启动时决定是否启用同步、realtime 和通知能力。

优点：

- 实现成本低，复用现有代码结构。
- 发布和部署链路改动较小。
- 主后端与业务后端仍共享同一套 HTTP 代码路径，维护成本低。

缺点：

- 角色配置填错时仍可能误启主后端能力。

### 4.2 方案 B：构建时拆分能力

通过 build tag 或单独构建目标，产出主后端与业务后端两套能力不同的产物。

优点：

- 能力隔离更彻底，业务后端无法误启同步能力。

缺点：

- 构建、发布、验证和文档都要维护两套路径。
- 当前仓库的收益不够高，复杂度明显上升。

### 4.3 结论

采用方案 A：同一套二进制 + 配置/启动参数切角色。

## 5. 角色模型

新增运行角色：

- `main`
  - 启用数据库同步
  - 启用远端 websocket 订阅
  - 启用本地 `/api/realtime/ws`
  - 启用通知逻辑
- `worker`
  - 禁用数据库同步
  - 禁用远端 websocket 订阅
  - 禁用本地 `/api/realtime/ws`
  - 禁用通知逻辑
  - 保留数据库连接与现有 HTTP API

兼容规则：

- 如果显式配置了角色，严格按角色执行。
- 如果未显式配置角色：
  - `http_api.port == 7778` 时默认视为 `main`
  - 其他端口默认视为 `worker`

这样可以兼容当前主实例默认跑在 `7778` 的既有部署习惯，同时避免新增业务端口时误启同步能力。

## 6. 后端职责拆分

### 6.1 启动阶段

在 `cmd/kanban-watcher/run.go` 统一解析角色，并据此生成能力开关：

- `EnableSync`
- `EnableRealtime`
- `EnableNotify`

后续所有初始化逻辑不再依赖端口硬编码，而是依赖角色能力。

### 6.2 main 实例

`main` 实例保留现有完整能力：

- 初始化数据库
- 启动 `sync.NewSyncService(...)`
- 构建 `realtime.NewHub()`
- 构建 `api.NewRealtimePublisher(...)`
- 注册 `/api/realtime/ws`
- 启动通知链路

### 6.3 worker 实例

`worker` 实例保留以下能力：

- 初始化数据库
- 构建查询相关 HTTP 路由
- 构建消息分发器
- 提供工作区、会话、消息、发送消息、队列、停止执行等 HTTP 接口

`worker` 实例禁止以下能力：

- 启动 `sync.NewSyncService(...)`
- 创建 `realtime.NewHub()`
- 注册 `/api/realtime/ws`
- 发送通知

## 7. 前端访问模型

前端链路拆成两条：

- HTTP 请求：继续访问当前页面对应的后端实例
- WebSocket 请求：统一访问主后端

### 7.1 HTTP

以下请求仍走当前 `baseUrl`：

- 工作区查询
- 会话消息查询
- 发送消息
- 队列状态查询
- 停止执行
- dev server 相关接口

### 7.2 WebSocket

以下实时能力统一连接主后端：

- `workspace_snapshot`
- `session_messages_appended`
- `workspace_view_updated`

前端不再默认把 websocket 地址拼接到当前 `baseUrl`，而是使用独立的 realtime 地址。

### 7.3 realtime 地址来源

`/api/info` 扩展一个 realtime 基地址字段，由后端显式下发：

- `main` 返回自己的 realtime 地址
- `worker` 返回主后端 realtime 地址

注意：当前 `/api/info` 是通过 `proxy.GetInfo(...)` 代理远端 `vibe-kanban` 的信息，因此这里不应破坏原有 `data.config.preview_proxy_port` 等字段结构。实现上应采用“保留原有代理返回，再在外层追加 watcher 本地字段”的方式，例如在 `data` 下增加本地扩展对象，而不是直接替换整个 `data`。

前端初始化时同时持有：

- `baseUrl`
- `realtimeBaseUrl`

其中：

- `src/lib/http-api.ts` 继续使用 `baseUrl`
- `src/lib/realtime-api.ts` 改用 `realtimeBaseUrl`

这样可以避免从 `16020` 打开的页面误去请求 `16020/api/realtime/ws`。

## 8. 路由与错误行为

### 8.1 `/api/realtime/ws`

- `main`：正常注册并提供 websocket 服务
- `worker`：不提供实时服务

`worker` 下有两种实现方式：

- 不注册路由，返回 404
- 注册路由但返回 503

本次推荐注册显式错误响应，返回 `503 realtime disabled for worker role`，便于定位部署与配置问题。

### 8.2 `/api/info`

在保持现有代理字段兼容的前提下，扩展返回字段：

- 当前实例角色
- realtime 基地址
- realtime 是否启用

这样前端和排障人员都能直接获知当前实例的运行模式。

### 8.3 通知逻辑

当前代码在 `runEventLoop` 中通过 `cfg.HTTPAPI.Port != 7778` 硬编码限制通知。

本次改为：

- 通过角色能力判断是否允许通知
- 不再依赖端口号

这样主后端端口未来即使变化，也不会破坏逻辑。

## 9. 配置设计

建议在配置中新增运行角色字段，例如：

```yaml
runtime:
  role: main
  realtime_base_url: http://127.0.0.1:7778
```

说明：

- `role`：实例角色，支持 `main` / `worker`
- `realtime_base_url`：主后端实时地址；对于 `main` 可省略并自动推导；对于 `worker` 可显式指定主后端地址

兼容策略：

- 未配置 `runtime.role` 时，根据 `http_api.port` 自动推导默认角色
- 未配置 `runtime.realtime_base_url` 时：
  - `main` 由当前监听地址推导
  - `worker` 默认回退到 `http://127.0.0.1:7778`

## 10. 日志与可观测性

启动时打印明确日志：

- `[Role] main: sync/realtime/notify enabled`
- `[Role] worker: http-only mode enabled`

这样即使出现误部署，也能在启动日志第一时间识别实例角色。

## 11. 风险与约束

### 11.1 误配置多个 main

本方案不会阻止运维把多个实例都配置成 `main`。如果发生这种情况，仍会出现重复同步。

本次仅通过以下方式降低风险：

- 默认角色自动推导偏保守
- 启动日志明确输出角色
- 配置文档强调“只允许一个 main”

### 11.2 主后端不可用

如果主后端不可用：

- 业务后端的 HTTP 查询和消息发送仍可正常工作
- 前端 websocket 退化失败，页面仅失去实时更新能力

这属于可接受的降级。

### 11.3 本地开发

本地开发中若只启动单实例，`7778` 默认仍是 `main`，现有主流程基本不受影响。

## 12. 测试策略

### 12.1 配置与能力单测

验证角色解析和默认值：

- 显式 `main` 时启用同步、realtime、通知
- 显式 `worker` 时禁用同步、realtime、通知
- 未配置角色且端口为 `7778` 时默认 `main`
- 未配置角色且端口非 `7778` 时默认 `worker`

### 12.2 后端单测

验证不同角色下的行为：

- `main` 会构建 realtime 路由
- `worker` 不会启动同步服务
- `worker` 请求 `/api/realtime/ws` 返回显式不可用响应
- `/api/info` 能返回角色与 realtime 地址

### 12.3 前端单测

验证地址分流：

- HTTP API 仍使用 `baseUrl`
- realtime API 改为使用 `realtimeBaseUrl`
- 未显式传入 realtime 地址时，能正确使用 `/api/info` 返回值或默认值

## 13. 实施步骤概览

1. 扩展配置结构，增加运行角色与 realtime 地址配置。
2. 在启动入口统一解析角色能力。
3. 让同步、realtime、通知全部改为按角色启停。
4. 扩展 `/api/info` 返回 realtime 相关字段。
5. 调整前端 realtime 连接逻辑，拆分 HTTP 与 websocket 地址。
6. 补齐后端与前端单测。
7. 更新启动和部署说明文档。

## 14. 预期结果

改造完成后：

- 只有主后端实例会同步远端数据并写库。
- 业务后端实例只提供基于同库数据的 HTTP 能力。
- 所有前端页面统一从主后端接收 websocket 实时数据。
- 数据库重复入库和多实例重复监听问题得到控制。
