# HA 卡片实时推送方案补充

## 背景

当前 Home Assistant 卡片已经切到本地 `kanban-watcher` HTTP API：

- 工作区列表：`/api/workspaces/active`
- 历史消息：`/api/workspaces/{workspace_id}/latest-messages`
- follow-up：`/api/workspace/{workspace_id}/follow-up`

现状问题：

- 工作区列表依赖定时刷新，不是实时状态
- 弹窗消息只能在打开时或发送后刷新，无法实时追加
- 一旦 `7777 -> 7778` 同步完成，卡片仍要等下一次 HTTP 拉取才能看到变化

目标：

- 工作区状态变化实时反映到卡片
- 弹窗消息支持实时追加
- HTTP 继续作为首屏加载与断线补偿机制
- WebSocket 失效时自动退回轮询

## 方案总览

采用：

- `WebSocket` 作为实时增量通道
- `HTTP API` 作为首屏加载和补偿通道
- `定时器轮询` 作为 WebSocket 断线时的降级方案

数据流：

1. 卡片初始化先走 HTTP 获取工作区列表
2. 用户打开某个 workspace 弹窗后，先走 HTTP 拉最新消息
3. 前端建立到 `7778` 的统一 WebSocket 连接
4. 后端在本地数据库写入新摘要/新消息后，向订阅客户端推送增量事件
5. 前端收到事件后局部更新列表或直接追加弹窗消息
6. WebSocket 断线时切回轮询；恢复后先补拉一次，再继续增量推送

## 后端设计

### 新增端点

- `GET /api/realtime/ws`

用途：

- 单一 WebSocket 连接承载实时工作区摘要变化和消息追加事件

鉴权：

- 复用现有 `api_key`
- 支持 `X-API-Key` 或 query 参数

### 推送事件

#### 1. `workspace_snapshot`

连接建立后的首个快照事件。

字段：

- `type = "workspace_snapshot"`
- `workspaces`

用途：

- 前端在连接建立后快速对齐一次当前工作区列表

#### 2. `workspace_updated`

某个 workspace 摘要变化时发送。

字段：

- `type = "workspace_updated"`
- `workspace`

包含字段至少有：

- `id`
- `name`
- `status`
- `latest_session_id`
- `has_unseen_turns`
- `has_pending_approval`
- `has_running_dev_server`
- `files_changed`
- `lines_added`
- `lines_removed`
- `updated_at`
- `last_message_at`

#### 3. `session_messages_appended`

某个 session 新增消息时发送。

字段：

- `type = "session_messages_appended"`
- `workspace_id`
- `session_id`
- `messages`

说明：

- `messages` 只发送增量新消息，不重发整段历史
- 消息结构与当前 HTTP `/latest-messages` 返回格式保持一致

#### 4. `ping`

保活事件。

字段：

- `type = "ping"`
- `ts`

### 服务端订阅与广播

新增一个轻量实时中心：

- 管理在线 WebSocket 客户端
- 记录每个客户端当前关注的 workspace/session
- 提供广播接口

订阅策略先做最小化：

- 所有客户端默认收到 `workspace_updated`
- 客户端打开弹窗后，发送订阅消息，声明当前关注的 `workspace_id/session_id`
- 只有关注该 session 的客户端才收到 `session_messages_appended`

### 与同步链路集成

集成点：

1. 工作区摘要同步完成后
- 对比摘要变化
- 若发生变化，广播 `workspace_updated`

2. 新消息写入数据库成功后
- 组装刚写入的增量消息
- 广播 `session_messages_appended`

### 一致性

HTTP 是权威通道，WebSocket 是增量通道。

规则：

- WebSocket 推送失败不影响数据库写入
- 前端可随时通过 HTTP 补拉恢复
- 同一消息前端按稳定键去重

建议前端去重键：

- 优先 `id`
- 如果未来需要，可退化为 `process_id + timestamp + content`

## 前端设计

### 初始化

1. 卡片加载时先拉 `/api/workspaces/active`
2. 建立 WebSocket 连接
3. 收到 `workspace_snapshot` 后，用快照纠正本地状态

### 弹窗

1. 打开 workspace 弹窗时
- 先请求 `/api/workspaces/{workspace_id}/latest-messages`
- 再向 WebSocket 发送订阅消息：
  - `action = "watch_session"`
  - `workspace_id`
  - `session_id`

2. 收到 `session_messages_appended`
- 若当前弹窗匹配该 session
- 则将新消息追加到当前消息列表底部
- 对重复消息做去重

### 断线补偿

WebSocket 断开时：

- 工作区列表回退到 5 秒轮询 `/api/workspaces/active`
- 当前打开弹窗回退到 3 秒轮询 `/latest-messages`

WebSocket 恢复时：

- 立即补拉一次当前列表
- 如果弹窗开着，再补拉一次当前消息
- 然后关闭降级轮询

## 分阶段实施

### 第一阶段

- 新增 `/api/realtime/ws`
- 支持 `workspace_snapshot`
- 支持 `workspace_updated`
- 前端工作区列表实时刷新

### 第二阶段

- 支持 `watch_session`
- 支持 `session_messages_appended`
- 弹窗消息实时追加

### 第三阶段

- 增加断线重连和补偿轮询
- 增加心跳和连接状态展示

## 验证点

1. `7777` 上游状态变化后，卡片列表能在实时通道里更新
2. 打开弹窗后，新消息能直接追加到底部
3. 关闭 WebSocket 后，轮询补偿仍能更新
4. 恢复 WebSocket 后，不会重复追加同一条消息
5. 发送 follow-up 后，弹窗内消息能通过实时事件回流
