# 前端反向代理接入-测试概述

## 测试目标

- 验证前端默认不再直连 `127.0.0.1:后端端口`
- 验证 `nginx` 能将 `/{frontend_port}/api/...` 转发到对应后端
- 验证通过 `/{frontend_port}/` 访问时，Vite 开发态资源仍可正常加载

## 自动化测试

执行命令：

```bash
npm test -- --run tests/playground.test.ts tests/web-entry.test.ts tests/build-config.test.ts
```

结果：

- `tests/playground.test.ts` 通过，确认默认 `baseUrl` 为空字符串，走相对代理
- `tests/web-entry.test.ts` 通过，确认 `/{port}/preview` 能识别为预览页
- `tests/build-config.test.ts` 通过，确认 Web 入口使用 `base: "./"`

## Nginx 校验

执行命令：

```bash
nginx -t
```

结果：

- 语法检查通过
- 已将仓库配置同步到 `/opt/homebrew/etc/nginx/servers/nginx-dynamic-proxy.conf`

## 本地联调

说明：

- 由于本机 `6020`、`6021` 已被占用，联调使用 `6022 -> 16022`
- 前端使用本项目 Vite 开发服务
- 后端使用最小 HTTP 服务模拟 `/api/info`

验证结果：

- `curl http://127.0.0.1:2453/6022/api/info` 返回 `{"ok":true,"via":"backend-16022"}`
- `curl -H 'Referer: http://127.0.0.1:2453/6022/' http://127.0.0.1:2453/@vite/client` 返回 Vite 客户端脚本
- `curl -H 'Referer: http://127.0.0.1:2453/6022/' http://127.0.0.1:2453/src/web-entry.ts` 返回前端入口模块
- `curl -H 'Referer: http://127.0.0.1:2453/6022/@vite/client' http://127.0.0.1:2453/node_modules/vite/dist/client/env.mjs` 返回 Vite 依赖模块

## 结论

- 单实例共享 `nginx` 可满足多套前端/后端代理需求
- 浏览器只需要访问前端入口，`/api` 与开发态静态资源都可由 `nginx` 正确转发
- 当前验证覆盖 HTTP 与 Vite 资源加载链路，WebSocket 依赖相同 upgrade 头策略，未单独做握手脚本验证
