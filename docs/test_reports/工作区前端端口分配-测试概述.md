# 工作区前端端口分配-测试概述

日期：2026-03-26

## 测试范围

- 固定管理 API `7778` 正常可用时，启动脚本通过接口获取前端端口
- 未显式设置 `KANBAN_API_KEY` 时，启动脚本可从配置文件读取 `http_api.api_key`，并正确去掉 YAML 引号
- 固定管理 API 不可用时，启动脚本回退到数据库兜底命令
- 本地端口缓存不存在时，`logs` 命令会查询数据库中的已分配端口映射
- 前端端口和后端端口推导规则保持一致：`backend_port = frontend_port + 10000`
- 前后端进程仍保持后台启动，后端 stderr 继续镜像到控制台与日志文件
- Go 侧端口分配服务、store、server 与兜底 CLI 编译通过

## 执行结果

1. `bash scripts/test_start_dev_frontend_port_api.sh`
   - 结果：通过
   - 说明：验证脚本会请求 `http://127.0.0.1:7778/api/workspace/{workspace_id}/frontend-port`，并使用返回的前端端口启动 Vite，同时用 `frontend_port + 10000` 作为后端端口。

2. `bash scripts/test_start_dev_api_key_from_config.sh`
   - 结果：通过
   - 说明：验证脚本在未显式设置 `KANBAN_API_KEY` 时，会从 `~/.config/kanban-watcher/config.yaml` 读取 `http_api.api_key`，并自动去掉 YAML 引号后再发请求。

3. `bash scripts/test_start_dev_frontend_port_db_fallback.sh`
   - 结果：通过
   - 说明：验证当管理 API 调用失败时，脚本会调用 `go run ./cmd/kw_frontend_port reserve --workspace <workspace_id>` 走数据库兜底分配，并继续按分配结果启动前后端。

4. `bash scripts/test_start_dev_backend_console_logging.sh`
   - 结果：通过
   - 说明：验证前后端仍为后台启动，脚本能快速返回，后端 stderr 仍会同步输出到当前控制台并写入后端日志文件。

5. `bash scripts/test_start_dev_logs_command_db_lookup.sh`
   - 结果：通过
   - 说明：验证当本地端口缓存不存在时，`logs` 命令会优先调用 `go run ./cmd/kw_frontend_port lookup --workspace <workspace_id>` 查询数据库中的已分配端口映射。

6. `bash scripts/test_start_dev_logs_command.sh`
   - 结果：通过
   - 说明：验证 `logs` 命令仍会持续跟随后端和前端日志文件。

7. `go test ./internal/service ./internal/store ./internal/server ./cmd/kw_frontend_port`
   - 结果：通过
   - 说明：验证前端端口分配器、store 层、server 接口以及兜底 CLI 编译与相关单测正常。

## 结论

当前方案已经满足：

- API 优先分配前端端口
- API 不可用时数据库兜底
- 无本地缓存时优先复用数据库中的端口映射
- 后端端口自动推导
- 已有启动脚本后台运行与日志行为未回归
