# 通知重复问题分析

## 问题描述
用户报告：任务在重复操作时仍然收到通知，且"开始注意取消后，计时点没有被置位"。

## 代码分析

### 关键发现

1. **UpdatedAt 变化会导致计时重置**
   - 当任务的 `UpdatedAt` 变化时（有新操作），会生成新的 `NotificationKey`
   - 新的 Key 会被视为新的"关注周期"，导致 `FirstSeenAt` 被重置
   - 这意味着如果任务一直在操作，计时会被不断重置，**不会**通知

2. **注意力切换时计时会被重置**
   - 当工作区从"需要关注"变成"不需要关注"时，所有跟踪记录被清除
   - 当再次变成"需要关注"时，`FirstSeenAt` 和 `ConfirmedAt` 都被重新设置
   - 这是**正确的行为**

3. **叠加提醒机制**
   - 发送通知后，在 `repeatInterval` 内不会重复通知
   - 只有超过 `repeatInterval` 后才会再次提醒

### 可能的问题场景

#### 场景 A：任务操作停止后触发通知（预期行为）
1. 任务开始需要关注，UpdatedAt = T1
2. 任务有新操作，UpdatedAt = T2（计时重置）
3. 任务有新操作，UpdatedAt = T3（计时重置）
4. 任务操作停止，UpdatedAt 不变
5. 从 T3 开始计时，达到阈值后通知

这是**预期行为**：任务操作停止后问题仍未解决，应该通知。

#### 场景 B：短时间内反复切换（可能的问题）
1. 工作区需要关注 → 设置 FirstSeenAt = T1
2. 工作区不需要关注（API 更新延迟）→ 清除记录
3. 工作区需要关注（问题仍在）→ 设置 FirstSeenAt = T3

如果步骤 2-3 之间的时间很短（比如 30 秒），用户可能认为"计时点没有被置位"，但实际上是被重置了。

#### 场景 C：叠加提醒被误解为"重复通知"
1. 发送第一次通知
2. 5 分钟后（repeatInterval），发送第二次通知
3. 用户认为这是"重复通知"

这是**预期行为**：叠加提醒是为了确保用户不会忽略问题。

## 建议的修复方向

### 方向 1：智能继承计时（可选优化）
当 UpdatedAt 变化但问题仍在时，可以选择继承之前的计时：
- 如果新 Key 与旧 Key 属于同一工作区
- 且切换时间很短（比如 < 1 分钟）
- 则继承旧的 FirstSeenAt

### 方向 2：更清晰的日志
添加日志记录计时重置的原因，帮助调试：
- 为什么计时被重置（UpdatedAt 变化 / 注意力切换）
- 当前的计时状态

### 方向 3：配置选项
允许用户配置：
- 是否在 UpdatedAt 变化时重置计时
- 叠加提醒的间隔

## 当前代码行为总结

| 场景 | 行为 |
|------|------|
| UpdatedAt 变化 | 计时重置（视为新周期） |
| 注意力从 true→false→true | 计时重置（清除后重建） |
| 已通知后仍在阈值内 | 叠加提醒（repeatInterval 后） |
