# 通知开关优化设计

## 背景

当前系统存在两个通知问题：
1. **多后端重复通知**：多个后端服务同时运行时，每个实例都会独立触发通知，导致重复通知
2. **启动时不必要通知**：项目启动时立即进行首次轮询并触发通知，但此时用户并未期待收到通知

## 目标

1. 只有主端口（7778）的后端服务才能发送通知
2. 启动后的首次轮询不触发通知

## 设计方案

### 1. 端口限制 - 只有 7778 端口可通知

**改动位置：** `cmd/kanban-watcher/run.go` 的 `handlePollResult` 函数

**实现逻辑：**
```go
func handlePollResult(..., cfg *config.Config, ...) {
    // 端口检查：只有主端口 7778 才能发送通知
    if cfg.HTTPAPI.Port != 7778 {
        // 非主端口实例，只更新菜单栏，跳过所有通知逻辑
        if trayApp != nil {
            trayApp.UpdateWorkspaces(result.Workspaces)
        }
        return
    }

    // 原有的完整处理逻辑（包括通知）...
}
```

**影响范围：**
- 企业微信通知：被限制
- 本地弹框通知：被限制
- 菜单栏更新：不受影响，所有实例都会更新

### 2. 首次轮询静默 - 启动时不通知

**改动位置：** `cmd/kanban-watcher/run.go` 的 `runEventLoop` 函数

**实现逻辑：**
```go
func runEventLoop(...) {
    isFirstPoll := true  // 新增标志：标记首次轮询

    for {
        select {
        case result := <-pollResults:
            if isFirstPoll {
                isFirstPoll = false
                // 首次轮询只更新菜单栏，不发送通知
                if trayApp != nil {
                    trayApp.UpdateWorkspaces(result.Workspaces)
                }
                continue  // 跳过后续通知逻辑
            }

            // 原有的完整处理逻辑（包括通知）...
        }
    }
}
```

**说明：**
- 首次轮询静默是独立于端口限制的逻辑
- 即使是 7778 端口，启动后的第一次轮询也不会通知
- 从第二次轮询开始，7778 端口正常通知，其他端口继续静默

## 测试验证

1. **端口限制测试：**
   - 启动 7778 端口实例，验证可以正常发送通知
   - 启动非 7778 端口实例（如 7779），验证不发送通知
   - 同时运行两个实例，验证只有 7778 发送通知

2. **首次静默测试：**
   - 启动服务后观察第一次轮询是否静默
   - 等待第二次轮询，验证正常通知（如果有需要关注的工作区）

3. **组合测试：**
   - 7778 端口启动后首次轮询静默
   - 7778 端口第二次轮询正常通知
   - 非 7778 端口所有轮询都不通知

## 风险评估

- **低风险**：改动集中在一个文件，逻辑简单清晰
- **向后兼容**：不影响现有配置和功能
- **可回滚**：如有问题可快速回退
