# 通知模块优化：稳定性确认机制

> **Goal:** 解决通知误报问题 - 用户已处理完毕但仍收到通知

> **Architecture:** 引入"稳定性确认"机制，要求 `NeedsAttention` 状态在连续两个轮询周期都成立才视为真正需要关注，避免 API 延迟导致的瞬时误报。

> **Tech Stack:** Go, kanban-watcher 项目

---

## 背景

### 问题描述
当前通知触发条件为 `NeedsAttention = HasUnseenTurns || HasPendingApproval`。这两个字段是**瞬时状态**，当用户处理完毕后，API 不会立即更新，导致通知基于过时的状态触发，造成误报。

### 设计目标
- **更稳**：接受最多 1 个轮询周期（30 秒）的延迟，但显著减少误报
- **简单**：每次需要关注的事件重新计时，不做智能继承

---

## 状态机设计

### 新增字段

在 `AttentionEntry` 中增加 `ConfirmedAt` 字段：

```go
type AttentionEntry struct {
    Key          NotificationKey // 工作区标识
    FirstSeenAt  time.Time       // 首次发现需要关注的时间
    ConfirmedAt  *time.Time      // 确认需要关注的时间（连续第二个周期）
    NotifiedAt   *time.Time      // 已发送通知的时间
}
```

### 状态转换

```
[首次发现 NeedsAttention=true]
         ↓
    设置 FirstSeenAt
         ↓
[下次轮询仍 NeedsAttention=true]
         ↓
    设置 ConfirmedAt ✓ 确认稳定
         ↓
[持续保持，计时等待阈值]
         ↓
[达到阈值] → 发送通知
         ↓
[问题解决 NeedsAttention=false]
         ↓
    清除跟踪记录
```

### 通知触发条件变更

| 版本 | 触发条件 | 最早通知时间 |
|------|----------|--------------|
| 原版 | `now - FirstSeenAt >= threshold` | 1 个周期（30秒） |
| 新版 | `now - ConfirmedAt >= threshold` | 2 个周期（60秒） |

---

## 数据流

### 轮询处理流程（ProcessWorkspaces）

```
1. 构建当前活跃工作区 ID 集合
         ↓
2. 清理已不存在的工作区条目
         ↓
3. 对每个工作区：
   a. 若 NeedsAttention=false → 清除跟踪记录
   b. 若 NeedsAttention=true 且无跟踪记录 → 创建条目，设置 FirstSeenAt
   c. 若 NeedsAttention=true 有记录但无 ConfirmedAt → 设置 ConfirmedAt
   d. 若 NeedsAttention=true 有记录且有 ConfirmedAt → 检查是否达到阈值
         ↓
4. 返回需要通知的工作区列表
```

## 持久化变更

### 新增字段持久化

`persistedEntry` 增加 `confirmed_at` 字段：

```go
type persistedEntry struct {
    WorkspaceID  string     `json:"workspace_id"`
    CompletedAt string     `json:"completed_at"`
    FirstSeenAt time.Time  `json:"first_seen_at"`
    ConfirmedAt *time.Time `json:"confirmed_at,omitempty"` // 新增
    NotifiedAt  *time.Time `json:"notified_at,omitempty"`
}
```

### 兼容性处理
- 旧版本状态文件无 `confirmed_at` 字段，读取时该字段为 nil
- 对于已有记录（首次发现时间 < threshold），需要重新确认
- 迁移策略：加载时将 `FirstSeenAt` 同时设为 `ConfirmedAt`，保持向后兼容

## 测试策略

### 单元测试覆盖

1. **首次发现 → 确认状态**
   - 模拟两个连续周期都返回 NeedsAttention=true
   - 验证 FirstSeenAt 和 ConfirmedAt 都被正确设置

2. **瞬时误报场景**
   - 周期1: NeedsAttention=true → 设置 FirstSeenAt
   - 周期2: NeedsAttention=false → 清除记录
   - 验证不会发送通知

3. **真正需要关注**
   - 周期1: NeedsAttention=true → 设置 FirstSeenAt
   - 周期2: NeedsAttention=true → 设置 ConfirmedAt
   - 周期3: 检查是否达到阈值

4. **问题解决后重新触发**
   - 问题解决 → 清除记录
   - 再次触发 → 重新计时

## 效果对比

| 场景 | 原版行为 | 新版行为 |
|------|----------|----------|
| 瞬时 API 延迟（30秒内恢复） | 可能误报 | 不会误报（需连续两个周期确认） |
| 真正需要关注 | 30秒后通知 | 60秒后通知（+1周期确认） |
| 用户处理后快速恢复 | 可能误报 | 不会误报 |

## 后续优化方向（本期不做）

1. **可配置确认周期数**：允许用户设置 1/2/3 个周期
2. **智能继承**：短时间内重复触发不重新计时
3. **通知冷却期**：发送通知后一段时间内不重复通知（当前已有基于 CompletedAt 的去重）
