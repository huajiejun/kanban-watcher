# Kanban Watcher 网页版工作区设计

Date: 2026-03-24

## Goal

为 `kanban-watcher` 新增网页版工作区主入口，并将原本的本地预览页迁移到 `/preview` 路由。

主入口页面不是替换 Home Assistant 卡片，而是在现有 API 和 UI 能力之上新增一个“菜单 + 内容”的网页容器：

- 左侧是项目状态栏，复用现有项目状态卡片
- 右侧是工作区内容区，复用现有点击后弹出的对话界面
- 右侧最多同时平铺 4 个工作区会话窗格
- 当任务从非 `需要注意` 状态变成 `需要注意` 时，自动在右侧打开对应工作区
- 用户手动关闭后，不再因普通刷新重复自动弹出
- 任务手动关闭后仍保留在左侧 `需要注意` 分组
- 多个已打开窗格需要并行存在，并随列表刷新同步更新

## Scope

本次设计只覆盖网页工作区入口及其前端交互，不改动后端 API 协议。

在范围内：

- 新增独立网页工作区页面
- 将现有预览页迁移到 `/preview`
- 提炼和复用现有卡片列表与会话内容渲染逻辑
- 支持右侧多窗格平铺展示
- 支持“自动打开”“手动关闭”的前端状态流转

不在范围内：

- 修改 Home Assistant 卡片对外配置格式
- 重新设计消息协议或实时协议
- 新增“忽略关注”类业务状态

## User Intent

用户希望把当前偏卡片/弹窗式的查看方式扩展成一个真正可持续使用的网页工作区：

- 左侧像菜单，稳定承载全量工作区状态
- 右侧像内容区，承载当前关注中的对话内容
- 重点任务进入 `需要注意` 时能主动推到眼前
- 用户关闭后界面要尊重人工判断，不能反复打扰
- 默认网页首页就是这个工作区入口，预览能力降到 `/preview`

## Existing Reuse Points

当前仓库已经有两块可直接复用的前端能力：

1. 工作区状态列表
   - 现有 `kanban-watcher-card` 已支持按 `需要注意 / 运行中 / 空闲` 分组渲染工作区卡片
   - 现有状态图标、相对时间、文件变更统计已经具备复用价值

2. 工作区对话详情
   - 现有点击卡片后弹出的对话视图，已经包含消息列表、工具调用详情、输入框、快捷按钮、发送/停止/排队等交互
   - 这些内容应从“模态弹窗容器”中抽离，复用到网页右侧内容窗格

因此本次不建议再造一套新 UI，而是拆分容器层与内容层。

## Information Architecture

页面采用固定双栏结构：

1. 左侧项目状态栏
2. 右侧工作区内容区

### 左侧项目状态栏

职责：

- 展示全部工作区的状态分组
- 作为导航入口，允许用户手动打开任意工作区到右侧
- 承载“需要注意”任务的可见性

内容结构：

- 顶部页面标题与刷新/连接状态摘要
- `需要注意`
- `运行中`
- `空闲`

分组规则沿用现有卡片逻辑，避免网页版和卡片版出现判定分叉。

### 右侧工作区内容区

职责：

- 展示当前已打开的工作区会话
- 承载自动弹出的关注任务
- 支持关闭窗格

右侧不再使用全屏模态遮罩，而是作为页面内内容区域常驻展示。

## Layout Design

### 整体布局

- 页面采用左右分栏
- 左栏固定宽度，适合作为菜单区
- 右栏占剩余宽度，适合作为内容区
- 移动端或窄屏时改为上下布局，但桌面端保持双栏为主

### 左栏视觉定位

左栏更像导航菜单，而不是看板列：

- 宽度固定
- 内容纵向滚动
- 保留分组标题与卡片列表
- 当前已在右侧打开的工作区需要有选中态或打开态提示

### 右栏视觉定位

右栏是工作区内容舞台，内部按打开数量自动分栏：

- 1 个窗格：占满右栏
- 2 个窗格：2 等分
- 3 个窗格：3 等分
- 4 个窗格：4 等分

所有窗格高度保持与右栏内容区一致，内部滚动而不是整页拉高。

不支持超过 4 个同时打开的窗格。

若出现第 5 个打开请求，规则统一为：

- 默认顶替当前右侧最早打开的那个窗格
- 不区分该窗格是手动打开还是自动打开
- 被顶替的窗格直接关闭，不保留在右侧
- 左侧列表不受影响

## Interaction Model

### 1. 手动打开

用户点击左侧工作区卡片时：

- 如果该工作区尚未在右侧打开，则加入打开列表
- 如果已打开，则将其设为当前活动窗格
- 手动打开一个当前处于 `dismissedAttentionIds` 的工作区时，应清除该关闭抑制状态
- 不再弹模态层

### 2. 自动打开

系统轮询或实时更新工作区状态时：

- 若某工作区此前不属于 `需要注意`
- 当前更新后进入 `需要注意`
- 且该工作区不在“手动关闭后抑制自动弹出”集合中

则自动加入右侧打开列表。

这是一个“状态跃迁触发器”，不是“只要在关注里就每次都打开”。

首次进入网页时不自动打开当前已经处于 `需要注意` 的历史任务，只记录当时的关注快照。从首次加载之后发生的新跃迁开始，才触发自动打开。

### 3. 手动关闭

用户关闭右侧某个窗格时：

- 该窗格从右侧打开列表移除
- 若该工作区当前仍属于 `需要注意`，则记录为“已手动关闭，不再自动弹出”
- 该工作区仍继续显示在左侧 `需要注意` 分组

### 4. 再次进入关注

若某任务已关闭过，但之后离开 `需要注意`，又再次从非关注转为关注：

- 视为新的状态跃迁
- 可以再次触发自动打开

这保证“关闭”抑制的是当前这一轮关注，而不是永久静默。

## State Model

网页工作区需要新增几类页面级状态：

- `openWorkspaceIds`
  - 当前右侧打开的工作区列表，顺序决定平铺顺序
- `activeWorkspaceId`
  - 当前活动窗格，用于高亮和定位
- `dismissedAttentionIds`
  - 当前处于关注态、但用户手动关闭后不再自动弹出的工作区
- `previousAttentionMap`
  - 用于比较本次和上次数据，判断是否发生“刚进入关注”

这些状态只属于网页入口容器，不应污染现有 Home Assistant 卡片主路径。

## Component Boundaries

推荐将现有 `kanban-watcher-card.ts` 内的 UI 能力拆成更清晰的责任边界：

1. 工作区列表渲染层
   - 负责分组、卡片显示、点击事件发出

2. 工作区会话内容层
   - 负责消息列表、工具消息、快捷按钮、输入区、发送区

3. 页面容器层
   - 负责网页双栏布局
   - 负责打开/关闭/自动打开等状态编排

4. Home Assistant 卡片容器层
   - 保留现有卡片外壳与模态交互

网页版和卡片版共享内容组件，但各自保留自己的容器和状态流。

## Data Flow

数据获取继续沿用当前 API 模式：

- 页面加载时请求活跃工作区
- 按现有机制建立实时连接或轮询
- 需要打开某个工作区时请求其最新消息
- 发送消息、停止执行、队列操作沿用现有接口

新增的数据处理逻辑集中在前端：

1. 拉取/接收工作区列表
2. 计算分组
3. 比对前后 `needs_attention`
4. 产出新的 `openWorkspaceIds / dismissedAttentionIds`
5. 同步刷新左栏和所有右侧已打开窗格

## Error Handling

### 页面级错误

- 左栏工作区加载失败时，显示清晰错误态
- 右栏某个窗格消息加载失败时，只影响该窗格，不阻断其他窗格

### 状态不一致处理

- 若右侧打开的工作区在最新数据中已不存在，应自动移除该窗格
- 若某工作区不再属于 `需要注意`，应从 `dismissedAttentionIds` 中清理，避免旧抑制状态泄漏到下一轮

### 容量上限处理

- 右侧最多 4 个窗格
- 第 5 个打开请求到来时，固定替换 `openWorkspaceIds` 中最早的一个
- 这个规则对手动打开和自动打开一视同仁

## Testing Considerations

实现时至少覆盖以下场景：

- 默认首页为网页工作区入口
- `/preview` 路由仍可访问原预览页
- 左栏分组与当前卡片版一致
- 点击左栏卡片后，右栏新增窗格
- 1 / 2 / 3 / 4 个窗格时布局正确
- 首次加载时，已有 `需要注意` 任务不会自动打开
- 某任务从非关注变成关注后自动打开
- 用户关闭关注任务后，不会因普通刷新再次自动打开
- 用户手动再次打开已关闭任务后，关闭抑制会被清除
- 同一任务离开关注后再次进入关注时，会重新自动打开
- 第 5 个打开请求到来时，会替换当前最早打开的窗格
- 多个已打开窗格会随工作区状态刷新和消息刷新同步更新
- 某个窗格消息接口失败时，其他窗格仍正常工作
- 移动端或窄屏下布局不破坏可用性

## Recommended Implementation Direction

推荐方案：独立网页容器 + 复用现有卡片与会话内容渲染逻辑。

原因：

- 最符合“左边菜单、右边内容”的页面心智
- 复用现有状态卡片和会话内容，改动集中且可控
- 不会把 Home Assistant 卡片逻辑和网页工作区逻辑继续耦合加深
- 为后续继续扩展网页工作区保留空间

不推荐本次采用“单一组件同时承担卡片和网页全部容器职责”的方案，因为它会把不同交互模型混在同一个大组件里，进一步放大已有复杂度。
