# Kanban Watcher 网页版工作区设计

Date: 2026-03-24

## Goal

为 `kanban-watcher` 新增一个独立的网页版工作区入口。

这个页面不是替换 Home Assistant 卡片，而是在现有 API 和 UI 能力之上新增一个“菜单 + 内容”的网页容器：

- 左侧是项目状态栏，复用现有项目状态卡片
- 右侧是工作区内容区，复用现有点击后弹出的对话界面
- 右侧最多同时平铺 4 个工作区会话窗格
- 当任务从非 `需要注意` 状态变成 `需要注意` 时，自动在右侧打开对应工作区
- 用户手动关闭后，不再因普通刷新重复自动弹出
- 任务仍保留在左侧 `需要注意` 分组；若用户显式标记为“忽略关注”，则保留可见但转为忽略态展示，不再参与自动打开

## Scope

本次设计只覆盖网页工作区入口及其前端交互，不改动后端 API 协议。

在范围内：

- 新增独立网页工作区页面
- 提炼和复用现有卡片列表与会话内容渲染逻辑
- 支持右侧多窗格平铺展示
- 支持“自动打开”“手动关闭”“忽略关注”的前端状态流转

不在范围内：

- 修改 Home Assistant 卡片对外配置格式
- 新增后端持久化字段来保存“忽略关注”
- 重新设计消息协议或实时协议

## User Intent

用户希望把当前偏卡片/弹窗式的查看方式扩展成一个真正可持续使用的网页工作区：

- 左侧像菜单，稳定承载全量工作区状态
- 右侧像内容区，承载当前关注中的对话内容
- 重点任务进入 `需要注意` 时能主动推到眼前
- 用户关闭后界面要尊重人工判断，不能反复打扰

## Existing Reuse Points

当前仓库已经有两块可直接复用的前端能力：

1. 工作区状态列表
   - 现有 `kanban-watcher-card` 已支持按 `需要注意 / 运行中 / 空闲` 分组渲染工作区卡片
   - 现有状态图标、相对时间、文件变更统计已经具备复用价值

2. 工作区对话详情
   - 现有点击卡片后弹出的对话视图，已经包含消息列表、工具调用详情、输入框、快捷按钮、发送/停止/排队等交互
   - 这些内容应从“模态弹窗容器”中抽离，复用到网页右侧内容窗格

因此本次不建议再造一套新 UI，而是拆分容器层与内容层。

## Information Architecture

页面采用固定双栏结构：

1. 左侧项目状态栏
2. 右侧工作区内容区

### 左侧项目状态栏

职责：

- 展示全部工作区的状态分组
- 作为导航入口，允许用户手动打开任意工作区到右侧
- 承载“需要注意”任务的可见性

内容结构：

- 顶部页面标题与刷新/连接状态摘要
- `需要注意`
- `运行中`
- `空闲`

分组规则沿用现有卡片逻辑，避免网页版和卡片版出现判定分叉。

### 右侧工作区内容区

职责：

- 展示当前已打开的工作区会话
- 承载自动弹出的关注任务
- 支持关闭窗格和忽略关注

右侧不再使用全屏模态遮罩，而是作为页面内内容区域常驻展示。

## Layout Design

### 整体布局

- 页面采用左右分栏
- 左栏固定宽度，适合作为菜单区
- 右栏占剩余宽度，适合作为内容区
- 移动端或窄屏时改为上下布局，但桌面端保持双栏为主

### 左栏视觉定位

左栏更像导航菜单，而不是看板列：

- 宽度固定
- 内容纵向滚动
- 保留分组标题与卡片列表
- 当前已在右侧打开的工作区需要有选中态或打开态提示

### 右栏视觉定位

右栏是工作区内容舞台，内部按打开数量自动分栏：

- 1 个窗格：占满右栏
- 2 个窗格：2 等分
- 3 个窗格：3 等分
- 4 个窗格：4 等分

所有窗格高度保持与右栏内容区一致，内部滚动而不是整页拉高。

不支持超过 4 个同时打开的窗格。

若出现第 5 个自动打开请求，规则统一为：

- 只允许淘汰“自动打开产生的窗格”
- 优先淘汰最早自动打开且当前不是活动窗格的项
- 不得挤掉手动打开的窗格
- 若当前 4 个窗格都来自手动打开，或唯一可淘汰项正是当前活动窗格，则本次不自动打开，仅保留左侧 `需要注意` 提示

## Interaction Model

### 1. 手动打开

用户点击左侧工作区卡片时：

- 如果该工作区尚未在右侧打开，则加入打开列表
- 如果已打开，则将其设为当前活动窗格
- 手动打开一个当前处于 `dismissedAttentionIds` 的工作区时，应清除该关闭抑制状态
- 不再弹模态层

### 2. 自动打开

系统轮询或实时更新工作区状态时：

- 若某工作区此前不属于 `需要注意`
- 当前更新后进入 `需要注意`
- 且该工作区不在“手动关闭后抑制自动弹出”集合中
- 且不在“忽略关注”集合中

则自动加入右侧打开列表。

这是一个“状态跃迁触发器”，不是“只要在关注里就每次都打开”。

首次进入网页时不自动打开当前已经处于 `需要注意` 的历史任务，只记录当时的关注快照。从首次加载之后发生的新跃迁开始，才触发自动打开。

### 3. 手动关闭

用户关闭右侧某个窗格时：

- 该窗格从右侧打开列表移除
- 若该工作区当前仍属于 `需要注意`，则记录为“已手动关闭，不再自动弹出”
- 该工作区仍继续显示在左侧 `需要注意` 分组

### 4. 忽略关注

对仍在 `需要注意` 中但用户暂不想被打扰的任务，提供“忽略关注”操作：

- 标记后，该工作区不再触发自动打开
- 左侧 `需要注意` 分组继续显示该项，但以“已忽略”弱化样式呈现
- 提供“恢复关注”操作，用户可撤销忽略状态

第一版建议采用前端本地状态，不新增后端字段。忽略状态在页面刷新后可以丢失，或后续再考虑本地存储持久化。

忽略与关闭抑制的优先级规则：

- `ignoredAttentionIds` 优先级高于 `dismissedAttentionIds`
- 标记忽略时，应同步清除对应 `dismissedAttentionIds`
- 恢复关注时，不立即自动打开；用户可手动打开，或等待下一次新的关注跃迁

### 5. 再次进入关注

若某任务已关闭过，但之后离开 `需要注意`，又再次从非关注转为关注：

- 视为新的状态跃迁
- 可以再次触发自动打开

这保证“关闭”抑制的是当前这一轮关注，而不是永久静默。

## State Model

网页工作区需要新增几类页面级状态：

- `openWorkspaceIds`
  - 当前右侧打开的工作区列表，顺序决定平铺顺序
- `activeWorkspaceId`
  - 当前活动窗格，用于高亮、定位，以及容量淘汰时避免误踢当前正在看的内容
- `dismissedAttentionIds`
  - 当前处于关注态、但用户手动关闭后不再自动弹出的工作区
- `ignoredAttentionIds`
  - 用户显式忽略关注的工作区
- `previousAttentionMap`
  - 用于比较本次和上次数据，判断是否发生“刚进入关注”

这些状态只属于网页入口容器，不应污染现有 Home Assistant 卡片主路径。

## Component Boundaries

推荐将现有 `kanban-watcher-card.ts` 内的 UI 能力拆成更清晰的责任边界：

1. 工作区列表渲染层
   - 负责分组、卡片显示、点击事件发出

2. 工作区会话内容层
   - 负责消息列表、工具消息、快捷按钮、输入区、发送区

3. 页面容器层
   - 负责网页双栏布局
   - 负责打开/关闭/自动打开/忽略关注等状态编排

4. Home Assistant 卡片容器层
   - 保留现有卡片外壳与模态交互

网页版和卡片版共享内容组件，但各自保留自己的容器和状态流。

## Data Flow

数据获取继续沿用当前 API 模式：

- 页面加载时请求活跃工作区
- 按现有机制建立实时连接或轮询
- 需要打开某个工作区时请求其最新消息
- 发送消息、停止执行、队列操作沿用现有接口

新增的数据处理逻辑集中在前端：

1. 拉取/接收工作区列表
2. 计算分组
3. 比对前后 `needs_attention`
4. 产出新的 `openWorkspaceIds / dismissedAttentionIds / ignoredAttentionIds`
5. 渲染左栏和右栏

## Error Handling

### 页面级错误

- 左栏工作区加载失败时，显示清晰错误态
- 右栏某个窗格消息加载失败时，只影响该窗格，不阻断其他窗格

### 状态不一致处理

- 若右侧打开的工作区在最新数据中已不存在，应自动移除该窗格
- 若某工作区不再属于 `需要注意`，应从 `dismissedAttentionIds` 中清理，避免旧抑制状态泄漏到下一轮
- 若某工作区已被忽略，但后端数据消失或变更，应在重新出现时重新判断
- 当同一工作区同时命中忽略和关闭抑制时，以忽略为准

### 容量上限处理

- 右侧最多 4 个窗格
- 自动打开超过上限时，只允许替换最早自动打开且当前不是 `activeWorkspaceId` 的窗格
- 若不存在可替换目标，则放弃本次自动打开，不影响现有手动打开内容

## Testing Considerations

实现时至少覆盖以下场景：

- 独立网页入口正常加载
- 左栏分组与当前卡片版一致
- 点击左栏卡片后，右栏新增窗格
- 1 / 2 / 3 / 4 个窗格时布局正确
- 首次加载时，已有 `需要注意` 任务不会自动打开
- 某任务从非关注变成关注后自动打开
- 用户关闭关注任务后，不会因普通刷新再次自动打开
- 用户手动再次打开已关闭任务后，关闭抑制会被清除
- 同一任务离开关注后再次进入关注时，会重新自动打开
- 标记“忽略关注”后，不再自动打开
- “恢复关注”后不会立即自动弹出，但下一次新的关注跃迁仍可触发自动打开
- 第 5 个自动打开任务到来时，不会挤掉手动打开的窗格
- 当 4 个窗格全部为手动打开时，新关注任务不会强制打开
- 某个窗格消息接口失败时，其他窗格仍正常工作
- 移动端或窄屏下布局不破坏可用性

## Recommended Implementation Direction

推荐方案：独立网页容器 + 复用现有卡片与会话内容渲染逻辑。

原因：

- 最符合“左边菜单、右边内容”的页面心智
- 复用现有状态卡片和会话内容，改动集中且可控
- 不会把 Home Assistant 卡片逻辑和网页工作区逻辑继续耦合加深
- 为后续继续扩展网页工作区保留空间

不推荐本次采用“单一组件同时承担卡片和网页全部容器职责”的方案，因为它会把不同交互模型混在同一个大组件里，进一步放大已有复杂度。
