# 工作区快捷网页入口设计

## 背景

当前工作区页面已经具备以下能力：

- 为工作区分配端口并启动开发服务
- 在工作区表头展示启动/暂停开发服务按钮
- 基于 `browser_url` 打开开发预览抽屉并通过 `iframe` 嵌入页面

后续需要补充一个更直接的快捷入口，用于从工作区表头一键打开对应的 FRP 地址网页。该入口需要同时覆盖桌面端与手机端，但两端的展示方式不同：

- 手机端要求全屏显示嵌入网页
- 宽屏端要求以较大的居中面板展示，不直接铺满整个视口

同时，交互位置需要与现有表头按钮保持一致：

- 图标放在“启动/暂停服务”按钮左侧
- 桌面端位于工作区表头
- 手机端位于弹出卡片的表头

## 目标

- 为具备可用 `browser_url` 的工作区提供快捷网页入口
- 点击后直接在当前页面内弹出 `iframe` 嵌入网页
- 手机端使用全屏弹层，宽屏端使用居中大面板弹层
- 不影响现有开发服务启动/暂停链路
- 不替换现有开发预览抽屉，保持能力分工清晰

## 非目标

- 不新增独立预览路由
- 不改动后端接口协议
- 不改变现有开发预览抽屉的触发方式与职责
- 不为无 `browser_url` 的工作区增加兜底启动逻辑

## 现状

现有实现中：

- [`src/components/workspace-conversation-pane.ts`](../../../src/components/workspace-conversation-pane.ts) 负责渲染工作区表头按钮
- [`src/web/workspace-home.ts`](../../../src/web/workspace-home.ts) 负责页面级工作区状态与开发预览抽屉
- 现有开发预览入口依赖 `showDevServerPreview` 和 `dev-server-preview-toggle` 事件
- `browser_url` 已用于生成嵌入预览地址，当前已有 iframe 预览能力

这意味着新功能更适合在现有“表头按钮 + 页面级弹层”架构上扩展，而不是新建第二套路由或独立页面。

## 方案对比

### 方案一：当前页面内新增快捷网页弹层

做法：

- 在工作区表头新增一个快捷网页按钮
- 点击后在 `workspace-home` 内打开新的页面级弹层
- 弹层内部使用 `iframe` 加载工作区 FRP 地址

优点：

- 交互最贴近“点击直接弹开”
- 桌面端与手机端可共用一套状态管理
- 改动集中在前端现有结构内，复用程度高

缺点：

- 需要额外维护一套弹层状态与样式

### 方案二：直接复用现有开发预览抽屉并拉伸为自适应大面板

做法：

- 新入口与现有开发预览入口共用同一弹层
- 通过响应式样式区分手机端和宽屏端尺寸

优点：

- 状态复用最多

缺点：

- “开发预览”和“快捷网页入口”语义混在一起
- 后续若两个入口需要并存不同交互，扩展性较差

### 方案三：跳转到独立网页容器页

做法：

- 点击后跳转到单独的网页预览路由
- 在新路由内全屏或大面板展示 iframe

优点：

- 页面职责清晰

缺点：

- 破坏当前工作区上下文
- 返回路径更重，不符合“快捷入口”的使用心智

## 结论

采用方案一：在当前页面内新增独立的“快捷网页弹层”。

原因：

- 最符合用户希望的“点击直接弹开”
- 可以严格满足手机端全屏、宽屏端适量大小的差异化要求
- 不会污染现有开发预览抽屉的职责

## 详细设计

### 1. 表头入口

在 [`src/components/workspace-conversation-pane.ts`](../../../src/components/workspace-conversation-pane.ts) 的表头动作区新增一个按钮：

- 按钮位于“启动/暂停开发服务”左侧
- 仅当工作区存在可用快捷网页地址时显示
- 使用独立事件，例如 `workspace-web-preview-toggle`
- 图标尺寸、圆角、hover/active 反馈与现有“文件入口”保持同一交互语言，作为视觉参考

表头从左到右的顺序调整为：

1. 文件入口
2. 快捷网页入口
3. 启动/暂停开发服务
4. 现有开发预览入口
5. 关闭

这样可以满足“图标位置放在启动和暂停服务的左侧”的要求，同时保留现有操作分组。

### 2. 页面级状态

在 [`src/web/workspace-home.ts`](../../../src/web/workspace-home.ts) 中新增一个页面级状态：

- `webPreviewWorkspaceId?: string`

用于标识当前打开快捷网页弹层的工作区。

配套方法：

- 打开弹层：记录工作区 ID
- 关闭弹层：清空工作区 ID
- 工作区关闭或数据消失时：同步清理该状态

### 3. 快捷网页地址来源

新入口复用现有工作区上的 `browser_url` 解析能力，不新增后端协议。

规则如下：

- 有可用 `browser_url` 时显示按钮
- 无 `browser_url` 时不显示按钮
- 弹层 iframe 的地址直接使用该工作区对应的嵌入网页地址

如果现有嵌入预览逻辑已经对本地地址和代理地址做过转换，则优先复用现有方法，避免重复实现 URL 规则。

### 4. 弹层布局

#### 手机端

- 弹层铺满整个视口
- 顶部显示标题栏
- 标题栏展示工作区名称与关闭按钮
- iframe 填满剩余空间

#### 宽屏端

- 使用半透明蒙层覆盖页面背景
- 中央显示大尺寸面板
- 面板不是全屏，保留外围边距
- 建议尺寸接近 `min(92vw, 1440px)` 宽、`min(88vh, 960px)` 高
- iframe 填满面板内容区

#### 通用关闭行为

- 点击右上角关闭按钮可关闭
- 点击蒙层空白区域可关闭
- 按 `Escape` 可作为可选增强能力，若实现成本低则一并支持

### 5. 与现有开发预览抽屉的关系

本次新增的是“快捷网页弹层”，不是替换现有开发预览抽屉。

两者职责区分：

- 开发预览抽屉：承载现有开发服务预览交互
- 快捷网页弹层：承载一键打开 FRP 地址的快捷入口

这样可以避免语义混乱，也便于后续分别调整交互。

### 6. 桌面端与手机端统一性

桌面端与手机端都复用同一个 `workspace-conversation-pane` 表头按钮定义。

由于手机端当前使用弹出卡片承载工作区内容，而该卡片复用了同一表头组件，因此新增按钮后：

- 桌面端会出现在工作区表头
- 手机端会出现在弹出卡片表头

无需为手机端单独再做一套按钮逻辑。

## 状态与边界处理

### 无地址

- 工作区没有可用 `browser_url` 时，不渲染快捷网页入口按钮

### 工作区关闭

- 如果用户关闭了当前工作区窗格，而该工作区正在展示快捷网页弹层，则同步关闭弹层

### 工作区数据刷新

- 如果工作区仍存在且地址仍有效，弹层保持打开
- 如果工作区在最新数据中不存在，自动关闭弹层

### 多工作区切换

- 同一时间只保留一个快捷网页弹层
- 用户从另一个工作区点击快捷网页入口时，直接替换当前弹层内容

## 测试策略

优先补充 [`tests/workspace-home.test.ts`](../../../tests/workspace-home.test.ts) 的行为测试：

1. 有地址时，表头渲染快捷网页按钮
2. 快捷网页按钮位于启动/暂停按钮左侧
3. 点击后，手机端渲染全屏弹层
4. 点击后，宽屏端渲染居中大面板弹层
5. 无地址时，不渲染快捷网页按钮
6. 点击关闭按钮后，弹层消失
7. 点击蒙层空白区域后，弹层消失
8. 关闭工作区时，关联弹层同步关闭

若样式判断在单元测试中不便直接验证，可通过：

- 断言弹层容器存在不同的模式标记，如 `data-layout="mobile"` / `data-layout="desktop"`
- 或断言不同 class 名称来确认布局路径

## 风险

### 风险一：现有预览与快捷网页入口职责混淆

如果按钮命名或图标不清晰，用户可能分不清“开发预览”和“快捷网页入口”。

缓解：

- 使用不同按钮图标与 `aria-label`
- 在代码结构上保持两个事件和两个容器独立

### 风险二：宽屏端面板过大或过小

如果仅靠固定像素值，可能在不同宽屏尺寸下体验不稳定。

缓解：

- 使用 `vw` 与 `vh` 上限控制
- 同时设置合理的 `max-width` 和 `max-height`

### 风险三：手机端 iframe 可用空间不足

如果标题栏或安全区处理不当，可能出现底部被遮挡。

缓解：

- 使用视口高度和安全区内边距
- 让 iframe 区域使用弹性布局自动撑满剩余空间

## 实施建议

实现阶段按以下顺序推进：

1. 先补测试，覆盖新按钮和弹层行为
2. 在 `workspace-conversation-pane` 增加按钮与事件
3. 在 `workspace-home` 增加页面级弹层状态和渲染
4. 补充响应式样式，区分手机端与宽屏端布局
5. 回归现有开发预览与关闭工作区流程，确认无回归
